#### 2. Build a gating multi-seed multi-candidate A/B harness with structured output that shares baseline runs
`loop` · tool: `ab.py`

Run each (policy, seed) pair as a subprocess parsing the score from stdout, accept one baseline plus MANY candidates per invocation (a repeatable --candidate mode sharing the baseline across {baseline,cand1,cand2,...} in one pass), use a bounded THREAD pool (workers only block on subprocesses, no Python CPU; ~3x faster, 22s vs 65s, capped because multi-second per-env init does not parallelize), and exit non-zero when any candidate's mean is strictly worse so it gates automatically. Sequential A/Bs against the substrate baseline and against the full policy each saturate the worker pool on their own baseline and cannot be parallelized as separate invocations, so the shared-baseline multi-candidate mode roughly halves wall-clock per winning cycle and enables cheap parameter sweeps without editing config constants between runs. Emit STRUCTURED machine-readable output (scores, exact commands, git SHA, dirty-tree state, env) with paired-bootstrap CIs, sign-test, trimmed means, and top-N wins/losses; human-only tables force manual transcription and block meta-analysis.

#### 3. Build a reusable in-process trace/diff tool for recurring regression questions; read the score first
`loop` · tool: `ab.py`

For any per-seed regression that persists unchanged across two or more ships, capture and DIFF per-tick per-agent traces (baseline vs current) to find the earliest divergence tick BEFORE forming another hypothesis -- a trace diff is far cheaper than another speculative A/B and repeatedly reveals mechanisms one step removed from the flag (a ~2-hour trace-and-diff tool build plus 20 minutes of analysis resolved a regression open across 15+ speculative experiments). A saved cogames scrimmage replay (--save-replay-dir) contains only static objects and aggregate stats, NOT per-tick frames, so to get per-tick behavior build an in-process trace runner driving mettagrid.Simulator directly (not the 'cogames play' subprocess) so you can grab each AgentPolicy._infos per tick (chosen skill, args, working memory, failures) plus sidecar state the policy never sees (hub inventory, agent positions, deaths); keep the A/B scoring harness as a subprocess for isolation but use the in-process path for trace analysis. It is much slower than a scrimmage, so read the SCORE first -- the failure is often obvious from the regressed score before the trace finishes. Build a reusable general capture+diff tool (skill histograms per role, hub-inventory time series, per-agent distance, death timelines, earliest-divergence tick) once you have three or more pending 'why does seed X regress' questions. When hosted logs 404 or are useless, build a deterministic local replay inspector: resim the BitWorld input+hashes from public artifacts and print final per-player state to cleanly separate 'died early' vs 'lost as crew' vs 'wandered until timeout' (in a temp dir to keep the repo clean) -- but note the tension: a thin resim-and-print inspector is worthwhile while rolling your own full replay PARSER is a dead-end detour, and if the real problem is the player's own tracing is too noisy/untargetable, fix the tracing source instead. Desired-but-unbuilt: ab.py --trace-seed N to auto-capture a trace for one interesting seed alongside the aggregate A/B, and trace_compare_agents.py for per-agent-pair trajectory divergence between two traces on the same seed.

#### 5. Serialize the one-time shared build/sync before fanning out parallel clients - parallel loaders corrupt the toolchain
`loop` · **negative result** · tool: `build_modulabot.py`

Run shared one-time setup serially in a single process BEFORE launching N parallel policy clients or bot subprocesses, then fan out. Parallel loaders race on building the shared native lib (e.g. libguidedbot.{so,dylib}) or on per-__init__ dependency extraction/sync when the pre-build step only guards 'build lib if missing' and not 'sync deps once'. Import POSIX-only modules like fcntl defensively (try/except ImportError) so the helper still loads on non-POSIX platforms.

#### 6. Intercept button-runners as an event-trigger en route, never as a standing positional bias
`crewrift` · tool: `button_runner_study.py`

A button-runner is just a high-value victim appearing predictably during the kill window. Trigger on the ROBUST signal (ticks_until_kill_ready + an observed approaching runner), not a fragile absolute tick like 'tick 500/900' (every meeting resets the clock); this reuses the window Search/Hunt already wake for, so no new scheduler is needed. Layer it as one pure helper (detect a runner over belief) plus small named bias hooks into Search positioning, victim selection, and Hunt strike-location -- each a preference, never a hard override of 'kill the visible isolated victim'. Empirically, runners travel SOLO (median 0 other crew within 48px) and 57.3% funnel through the Hydroponics-side mouth of the Bridge corridor (chokepoint ~x270-368), but ~42% are already at the bridge and not interceptable; bias search toward the corridor/Hydroponics, prefer cells ~120-320px off the button (peak ~220) on common approach paths, and score candidate cells as (pass-through fraction) x (distance band) x 1/(1+median_crew_near). Kill the runner en route then vent away, not at the button (the bridge is the most-trafficked room where a body is reported fast).

#### 8. Run the A/B harness as one subprocess per (policy, seed); render off, fail-fast, thread-parallel
`crewrift` · tool: `cogames play`

Run the A/B harness as one subprocess (cogames play) per (policy, seed) pair rather than importing the simulator in-process: this avoids cogames init-ordering concerns and prevents one crashed run from contaminating the rest. Use threads (not processes) since workers block on subprocess I/O; expect ~3x speedup bounded by a ~1.5s serial per-subprocess cogsguard init tail. Fail-fast on any worker exception (cancel remaining work and re-raise) — partial A/B results are worse than none. Force render off in batch runs (-r none) for ~25x speedup (a 6-seed run dropped from ~62s to ~2.5s) because the harness defaults render to gui even without a replay file.

#### 10. Recover Crewrift events black-box by re-simulating the replay and diffing pre/post-step state, not by parsing bytes or patching sim.nim
`crewrift` · ⚠ _session-derived, unverified_ · tool: `crewrift replay event reporter (re-sim diff)`

The robust way to derive Crewrift game events from a replay is to re-simulate with the real Crewrift sim and snapshot-diff two consecutive states, reusing the sim's already-exported procs; do NOT parse bytes and do NOT modify sim.nim or anything outside the reporter directory. Use the built-in ReplayPlayer loop as the canonical consumer template (loadReplay(path) -> initReplayPlayer(data) -> looping=false, mismatchQuit=true -> while replay.playing: replay.stepReplay(sim); stepReplay applies replay joins/leaves/inputs/chats, steps the real sim, and validates hashes), not a hand-rolled driver. gameHash() (sim.nim:1551+) hashes essentially the entire SimServer/Player state every tick and playback validates against it, so after each stepReplay the re-simulated state is provably identical to the original with exact tick alignment. Capture the pre-step snapshot AFTER that tick's joins/leaves/inputs are applied, not before, or player-index diffs are misaligned on leave ticks. Resolve tick-semantics disputes against the actual sim.step order, not intuition: within a tick this tick's input is applied, then movement/vent/cooldown update, then fresh-A kills fire from the UPDATED state (the naive 'press A, tick ends, kill applies from last tick's position' model is wrong); the Playing-phase per-player loop (sim.nim:3927-3948) does NOT break on a mid-loop phase change, and applyInput movement (2875) precedes the attack check (2887), so every player's tick-T movement still lands and the post-step snapshot holds correct positions. Events cleanly recoverable by state-diff: death/body (postStep.bodies.len > preStep.bodies.len; new Body color/slotId), task completed (tasks[t].completed[p] flips), meeting started (phase Playing->Voting), vote cast (voteState.votes[i] changes), ejection (voteState.ejectedPlayer), roles (players[i].role any tick after Lobby), emergency button (players[i].buttonCallsUsed increments). More generally, detect lifecycle/terminal events by diffing state across ticks when the protocol sends no explicit messages (an origin coordinate reset = new active object; a sudden stack-height drop = clear/score), keeping cross-tick diffing in a dedicated telemetry object separate from decision logic. SimServer.gameEventLoggingEnabled only controls the sim's own prose event logs, not core replay semantics. src/crewrift/replays.nim is the VIEWER/playback + hash-checking module, NOT the writer; the write path is the live server (crewrift.nim passes a temp .bitreplay as saveReplayPath into runServerLoop; server.nim opens replayWriter and writes leaves/joins/input-mask changes/tick hashes, then uploads on shutdown).

#### 12. Feature-flag every version difference behind a byte-identical flags-off substrate, freeze each ship as a snapshot, and trace regressions against the frozen config plus the git SHA
`loop` · **negative result** · tool: `experiments/regression_check.py`

Encapsulate every version-to-version behavioral difference as a feature flag on a single policy class, and guarantee all-flags-off is BYTE-IDENTICAL to the upstream starter policy on fixed seeds. Verify this substrate equivalence automatically (experiments/regression_check.py) as a HARD GATE at the start of every session and before every A/B, never after; it proves the only changed variable is the flag under test. If it fails (non-zero exit) the harness is contaminating the baseline — treat every A/B as invalid until fixed; new subsystems (a memory store) must stay empty and gated when flags are off; compile-time flags (Nim `when defined(...)`) work the same way. Re-run the cheap substrate check whenever a non-flag-gated shared path (perception, interpretation, pipeline wrapper, core navigation, decide.py, pursue_target.py) drifts, because otherwise every 'versus bare substrate' A/B is silently invalidated and the symptom is misattributed to the candidate flag; flag-gated code is safe to refactor as long as the flags-off path is untouched.

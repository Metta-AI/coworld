#### 1. Treat every doc name, claim, and structural survey as unverified until re-diffed against live source
`loop`

Every path, command, client-method, request-model name, behavior claim, parser vocabulary, pipeline-diagram step, and enumerated API/export list in a player's docs (PLAN/DESIGN/README/AGENTS.md/COOKBOOK/reviewer feedback) is an unverified hypothesis: ls the path, grep the symbol, walk the actual code's step order, source the action/observation vocabulary from the live parser, and diff against current source before trusting it, because players/bots/docs get moved or renamed often (crewborg's AGENTS.md pointed at COWORLD_README.md under packages/coworld/src/coworld/ when docs had moved to packages/coworld/docs/; a referenced scripts/capture.py belonged to a different player; a README promised mode completion/stalling decide() never implemented; an audit found docs listing trace-adapter names absent from code, omitting real modules like coworld_json_bridge.py and classes SharedMemoryView/AsyncStrategy/StrategyRunner, and listing events the SDK never emits; a pipeline diagram omitted a config-gated role-rebalancing step). Run a separate audit pass per error class (naming, paths, structure, protocol, status) rather than one generic pass, and re-read docs before finalizing because they drift on disk mid-session (a diagnoser design doc grew from 12 to 103 lines between two reads). When porting or restructuring from a fast-moving source, your initial structural map goes stale, so re-diff EACH module against live source as you port it (two parity bugs: a sub-state record missing fields the source had since added after it grew ~93 lines, and ported threshold logic that dropped an early-out) and copy the source's NAMED tuning constants (Crewmate*-style) rather than inlining literals, which silently breaks parity. Cross-check the live working tree against git status, not a memory note or README: a dirty checkout can list policies (modulabot, old cogames wrappers) only as deleted tracked files gone from disk, and 'framework exists' is not 'framework is intact' -- a live run failed because a cogames-agents checkout had many tracked files deleted yet still imported partially. Distinguish 'present and usable now' from 'tracked upstream / deleted locally'. For recovered/legacy tools, read actual imports and asset-path logic, not the header (one claimed PYTHONPATH=among_them was needed when the script self-located via Path(__file__).resolve().parent.parent, and listed Pillow though only pygame and numpy were imported).

#### 2. Resolve doc contradictions by reading authoritative source, never by averaging; never document phantom limitations; then fix the code gap
`generic` · **negative result**

Source is authority on current behavior. Resolve contradictions between docs, sibling docs, or independently-generated drafts by reading the authoritative game-engine/SDK source and picking the correct answer, never by averaging or hedging; prefer the draft carrying concrete values, types, file paths, and function names (one reporter's DESIGN.md said the event-log `player` column was int16 while its sibling README said int64, the canonical value). For Coworld role work the authoritative contract lives in metta (e.g. reporter contract at packages/coworld/src/coworld/docs/roles/reporter.md; cross-check docs/roles/<role>.md, COWORLD_README.md, types.py for schema, GAME_RUNTIME_README.md for env contracts, certifier.py for what validation runs) and a player repo's READMEs merely restate it. Do NOT document limitations or fallback paths the agent never exercises: a DESIGN.md falsely listed 'missing JSON fallback' as a P0 limitation, but the agent never serves the JSON protocol, so a non-binary first message would be a runner misconfiguration, not an agent gap -- phantom limitations send future work in the wrong direction. Maintain separate living docs (intended-behavior design, a source-verified impl plan with a dated current-gap list, migration docs) so later phases never silently depend on undelivered features.

`loop` · ⚠ _session-derived, unverified_

#### 19. Prove a 'merged' stacked PR actually reached main before deleting the branch
`generic` · ⚠ _session-derived, unverified_

GitHub's MERGED label and gh's state field lie about Graphite stacks: a Graphite squash-merge can show state CLOSED with mergedAt null, and a stacked PR whose base is another stack branch (not main) merges its squash commit into that base, so if an earlier sibling was squash-merged to main first, the later PR lands on an orphaned base and its work silently never reaches main. Before deleting a 'merged' branch, prove the code is on main: `git merge-base --is-ancestor <branch-tip> origin/main` (expect yes), confirm a branch-new file exists in origin/main, and check the two-dot diff origin/main..<branch-tip> is empty. To recover work that never reached main, KEEP the branch (its tip holds all the work) and either open a fresh PR from it targeting main directly or cherry-pick the tip onto a new branch off current main; do not delete the branch first.

#### 20. Graphite/PR mechanics: gt track parents, point origin at GitHub, dry-run, protect un-pushed commits, diff huge PRs per-file
`generic` · **negative result** · ⚠ _session-derived, unverified_ · tool: `gt (Graphite CLI)`

Chaining PR base branches on GitHub does NOT make them a Graphite stack: Graphite tracks parents in its own local metadata, so register them with `gt track` (metadata-only, safe), and clear bogus 'needs restack'/'N ahead' state by resetting local main to the GitHub commit (preserving local-only commits on a side branch first). `gt submit` pushes/fetches against the 'origin' remote regardless of per-branch upstreams and infers repo owner/name from origin's URL, so a local-clone origin makes submit unreachable (GitHub API 404) and submit refuses when local trunk main does not match origin/main -- rename the local-clone remote and point origin at GitHub before submitting (or override owner/name in .git/.graphite_repo_config), and move work commits onto the feature branch / realign main to origin/main. Run a `gt submit` dry-run first to catch stale metadata; `gt create` branches off the currently-TRACKED branch in metadata, not your last checkout, so verify and correct the stack parent after creating; `gt modify` is version-fragile (rejected --no-ai, choked on repeated message args) so fall back to a non-interactive `git commit --amend` then let `gt submit` update the PR.

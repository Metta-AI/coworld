# Harness, SDK & wire protocol — guide

Reference notes, design rationale, and negative results (34) for harness, sdk & wire protocol. Read when working in this area.

#### 1. Treat live schema and engine source as authority over the local checkout; map drift structurally
`loop` · ⚠ _session-derived, unverified_

Treat the live OpenAPI dump and running backend as authority for request bodies, not a local checkout, because checkout schemas drift (canonical world version, accepted fields); use the checkout to understand the algorithm but pull the live schema before building API calls, and when the typed CLI crashes fall back to the raw API (server now wraps episode rows in entries/an EpisodeRequestPage while the local CLI expected a bare list; some routes like /jobs/{job_id} existed in source but were absent from the OpenAPI dump). Distinguish 'empirically true' from 'mechanistically true' and verify against engine source, not client docs: a claim documented in 7 places across 3 files ('the local harness always assigns the policy to slot 0, always crewmate') was wrong at the mechanism level -- the server assigns roles via a seeded Fisher-Yates shuffle and slot 0 got crewmate only because the default seed (42) happened to shuffle that way. When a pinned vendored engine breaks an import, do NOT blindly rename to the 'did you mean X?' symbol; that often hides a structural class split (one monolithic config split into a connection/runtime object with binary_path/host/port and a pydantic *EnvConfig with num_players/max_ticks/seed, typically extra='forbid'). Trace the new module structure and map every old field before editing; minimize blast radius by returning the narrowest object callers use; a new keyword-only param with a default is backward-compatible, so verify keyword-only-with-default before assuming breakage. Build first to baseline engine-API drift before adding a feature, and when a build error sits in code you didn't touch, confirm it predates your change (check a sibling player) before assuming you broke it.

#### 2. Derive RULEBOOK/GAME_API from engine source; protocol capabilities are asymmetric and porting math silently diverges
`crewrift` · ⚠ _session-derived, unverified_

Before writing any agent for a coworld, derive two reference docs straight from engine source: a RULEBOOK (rules, roles, phases, win conditions) and a GAME_API (run the server, connect over WebSocket, interpret input frames, send actions). Verify protocol assumptions against the server's routing code, not docs, because send-path and read-path capabilities can be asymmetric: an agent in a private chatroom cannot READ global chat (server routes chat by an inChatroom flag and the chatroom view never renders global messages) yet can still SEND a global message via the chat packet without opening the UI -- check both directions. When a local harness pins a player's role, trace it to the server: client CLI arg -> a slots config field -> JSON -> --config:<json> -> server pins slot 0; a role-only pin does NOT mark the slot restricted (restriction keyed on name/token), and role parsers often accept multiple spellings case-insensitively ('crew'/'crewmate', 'imp'/'imposter'/'impostor'). When porting projection/perception math between languages, implementations silently diverge: a Python perception did proper ray-clipping from player to icon against the viewport while the Nim port merely clamped the icon's screen position to bounds; diff a ported geometry helper against the reference and confirm which matches the server's actual placement algorithm before relying on it.

#### 3. Keep the perceive->belief->mode boundary clean: modes emit scratch signals, never mutate belief; reject the delta pattern
`loop` · **negative result** · ⚠ _session-derived, unverified_

Keep deterministic mode logic free of direct belief mutation: have modes emit only scratch signals (e.g. a failed-strike observation) and merge them into belief in a single envelope/belief-merge step, routing any new self-correcting signal through that existing path, so behavioral inference stays traceable and modes stay pure. Keep the strategy-facing belief object clean -- never stash raw transport state or pending trace events on it -- and keep core decision functions pure and sink-free by routing emission through a separate emitter/hook rather than threading a sink into every signature. NEGATIVE: do NOT adopt a delta/patch mutation pattern (every hook/mode/task returning a BeliefStateDelta) unless you have real concurrency or time-travel needs; its only real benefit is merging independent concurrent writes, which does not apply when execution is sequential. Prefer direct mutation of mutable state and get traceability from a thin logging proxy, because defining the delta language (deep vs shallow vs path merges, append-vs-replace, conflict resolution) is hard and exists only to avoid a direct assignment. Keep explicit atomic 'message' boundaries only where data genuinely crosses a concurrency/async seam (an async decide loop publishing a deep-copied (ModeDirective, belief_updates) tuple to the fast act loop); do not generalize that one serialization point system-wide.

#### 4. Belief state: flexible dict, namespaced inferences, world knowledge vs task scratch, reset clears ad-hoc attrs
`loop` · ⚠ _session-derived, unverified_

Make the policy's belief state a flexible nested dict with a small set of required fields (current_task, position, role, perception outputs) and everything else free-form, so each mode stashes custom data and new behaviors add without schema migrations. Give LLM-injected/outer-loop inferences their own namespaced sub-dict (an `inferences` field modes read) and keep framework-managed fields (tick, position, view, occupancy_grid) read-only to the outer loop so an untyped dict's top-level assignment cannot clobber machine-perceived state. Store persistent world knowledge in belief state and only per-task control state in task-scoped memory: 'areas I've visited' is world knowledge (reuse viewport-confirmed occupancy-grid cells) and must live in belief, because task-scoping loses it on any interruption and causes re-exploration. A reset() must clear ad-hoc attributes stashed dynamically at runtime (delattr the unknown attributes), not only declared fields, or transient state leaks across episodes -- and a snapshot-based reset must include the bookkeeping that drives it (ActionMemory.clear() omitted the _standard_fields registry attribute itself, wiping the very list that tracks what to clear). Maintain a documented registry of every extension/scratch key an agent stores on shared state (belief_state.ext), recording each key's type/lifecycle/purpose to prevent namespace collisions, distinguishing keys that persist across ticks from keys in a wholesale-replaced dict (which every returning code path must re-include -- a fragile pattern), and prefer persistent storage for invariants like hysteresis bookkeeping. In a coworld policy beliefs in wm.beliefs persist across ticks and are the right place for a per-miner lock like locked_mining_tags; clear it when cargo returns to zero.

#### 5. LLM calls must NEVER block the real-time frame loop; async dispatch, per-kind timeouts, drop stale/late results
`loop` · **negative result**

Make 'the per-frame control loop never blocks on an LLM call' the top hard constraint for any LLM-driven game policy. A 5-9s blocking call inside the per-frame loop gets the bot dropped even when the call succeeds: while it blocks the server keeps streaming frames, the client's socket receive buffer overflows (~170 frames > 1MB vs typical 64-256KB OS buffers), the server's send fails and drops the player, and the next receiveMessage hangs forever (symptom: bot freezes repeating its last action). 'Another bot blocks and votes fine' is no proof: compare per-call duration to socket send-buffer headroom, not sync-vs-async (a ~1-3s in-process libcurl call survives; a ~5-9s fork+exec subprocess does not); blocking only survives confined to an idle phase (the voting interstitial). Budget ~50 LLM calls/game at ~2-4s each and dispatch asynchronously, consume the result on a later frame, allow only one pending output at a time, and if it misses the per-turn budget (~4s listen window) fall back immediately and DISCARD the late result. For many agents dispatch each as a non-blocking future via a ThreadPoolExecutor sized to the agent count and gather at a wall-clock deadline (MODTALKS_LLM_DEADLINE_SECONDS default 12s), leaving slow futures IN FLIGHT (providers rarely honor cancellation) -- measured: 8 concurrent Bedrock agents cut p50 ~33s to ~9.2s. Set per-call-kind timeouts (hypothesize/strategize 20s, react 15s, accuse/persuade 10s); retry ONLY transient failures (429, APITimeout, APIConnection, 5xx) with bounded backoff (max 2 retries 0.5s/1.5s), never deterministic 4xx, and abandon a retry whose backoff exceeds the deadline. Discard stale responses by comparing dispatch-time stage against current stage. Enforce context size with progressive multi-tier trimming dropping lowest-value sections first (halve sightings, then chat, then full chat log, then prior meetings, evidence scores LAST), newest-first. NEGATIVE: provider latency can structurally EXCEED the per-turn budget (Bedrock p50 ~9s vs a ~4.2s vote window) so the async result is almost always too late -- async dispatch alone does NOT solve latency; widen the budget or rely on the fallback. Measure provider p95 against the phase tick budget on the REAL production provider (cold live Bedrock), not a warm dev harness. Capture LLM context byte count at DISPATCH time (capturing at response time reads zero because dispatch may have cleared the buffer).

#### 6. Audit an LLM behavior menu across four axes and find the deterministic shadow policy that really decides
`generic` · **negative result** · ⚠ _session-derived, unverified_

When auditing behaviors exposed to an LLM, check four distinct axes per behavior: nominal (registry/parser knows the name), advertised (appears in the system prompt), really available (a runtime path can activate it and it passes the legality guard), and does-something (its decide() emits real actions, not a no-op). A mode can pass the first three and still be a pure no-op; an advertised no-op (e.g. an 'alibi_building' mode ending in noOpIntent()) is worse than an absent one because the LLM picks it and wastes turns. The legality guard and the prompt description are independent sources of truth that silently diverge (one mode was legal in code for any alive non-ghost including imposters while the prompt claimed crewmate-only). Even when telemetry shows directive_source=llm 73-93%, real authority can live in default mode-per-role selection and reflex transitions, so locate every strategic choice the LLM did not make before claiming it controls the agent.

#### 7. Select the LLM provider explicitly from env; a missing key is not 'LLM disabled'; copy a working sibling's config
`loop` · **negative result** · ⚠ _session-derived, unverified_

Make provider selection explicit and env-driven with a fixed first-match precedence so the same code runs local and hosted: disable-flag => deterministic ranker only; USE_BEDROCK / CLAUDE_CODE_USE_BEDROCK=1 truthy => AnthropicBedrock with platform creds (AWS_ACCESS_KEY_ID/SECRET/REGION); ANTHROPIC_API_KEY set => first-party Anthropic for local testing; else => scripted/deterministic only. NEGATIVE: a guidance provider hardcoded to the Anthropic Messages API treated a missing ANTHROPIC_API_KEY as 'LLM off' and silently ran with no LLM in an environment that actually had AWS Bedrock Claude credentials. An enable flag being set does not mean a call happens -- provider availability is a separate gate -- and verify the high-value path actually executed by looking for guidance/strategy traces, not merely that the policy ran without erroring. Boolean-ish flags are easy to misread (GUIDED_BOT_LLM_DISABLE=0 means do-NOT-disable; only 1/true/yes/on count true; a baked-in default-on image env like CREWBORG_LLM_MEETINGS=1 re-enabled the feature on the wrong provider): verify against the parser, audit every place the feature can be enabled, never assume 0 means off. When adding an LLM brain on a shared platform account, copy the model id, region, and client config from a known-working sibling rather than guessing: a proven config is AnthropicBedrock, model us.anthropic.claude-haiku-4-5-20251001-v1:0 (the us. cross-region inference profile), AWS_REGION=us-west-2, temperature 0, ~160 max_tokens, tight timeout. For a high-volume daily league, default to the cheapest capable LLM (Haiku 4.5, temp 0, capped tokens, hard per-call timeout) and reserve larger models behind an env override, because the league plays many episodes so cost/latency dominate.

#### 8. Verify the LLM path works inside the deployed container; confirm the backend from runtime logs, not config
`loop` · **negative result** · ⚠ _session-derived, unverified_

An LLM policy that works locally can silently no-op in a hosted pod because credential resolution differs by host. A meeting LLM failed in production because the resolver knew only local CLI/env keys and ECS task-role paths, while the host scheduled the pod under EKS IRSA (web-identity token file + AWS_ROLE_ARN/AWS_WEB_IDENTITY_TOKEN_FILE), yielding no_key -- and missing hosted creds are a SILENT degrader (the policy falls back to weaker heuristics, not a crash). Confirm the active backend from runtime logs, not intended config: a bare model id in logs (vs a us.anthropic.* inference profile) means Bedrock was never selected even with a 'Bedrock backend' commit, because USE_BEDROCK never reached the player pod; a 404 'Model use case details have not been submitted' from the direct Anthropic API means the account is gated, not a code bug, and tuning prompts changes nothing -- optimize the fallback path or upload without the LLM flag. Read the exact exception type/message to localize the backend: a direct Anthropic Messages API 404 is NOT a Bedrock failure and proves the player used direct Anthropic when Bedrock was intended. To isolate an env-injection bug to the right layer, check whether the per-job secret bundle actually contains the flag: it contained USE_BEDROCK=true for every slot yet the player still used direct Anthropic, proving the break was downstream of bundle creation (the runner not reading POLICY_SECRETS_URI or not merging that env into the player pods). Mitigation: resolve credentials for the mechanism the host uses and emit a startup llm_init trace recording the selected provider and present auth env vars, so a no_key failure self-diagnoses from episode logs.

#### 9. Honor the dispatcher submit() contract on the synchronous path; suspect result-plumbing when the LLM connects but emits nothing
`generic` · **negative result** · ⚠ _session-derived, unverified_

When a dispatch interface has both async-polling and synchronous paths, honor the submit() return contract on BOTH. A dispatcher built around async polling left the sync HTTP path half-wired: submit() computed a valid completion, stored it in a local, discarded it, and returned false, which the caller read as an error. A synchronous result needs a place to live (a syncResult/syncReady slot the poll step checks) or it is silently lost. When an LLM provider connects at startup but no output appears, suspect the dispatch/result-plumbing layer, not auth or the API call: verify the result is delivered to the consumer, not merely that the call was made. If a confirmation that LLM/reflex layers are even active is needed, check the logs (all decisions carrying directive_source:'default' means pure scripted play, dormant reactive/LLM layers).

#### 10. Nim/ORC: never share GC state across threads; channels never delivered in the bot binary -- use subprocess polling or a Unix-socket JSON protocol
`loop` · **negative result**

In threaded native players (Nim under --mm:orc), global mutable state (cached API key, connection pools, history seqs) is a GC-safety and data-race hazard the moment a worker thread touches it, and a {.gcsafe.} pragma alone does not fix it. NEGATIVE: Channel.tryRecv under ORC NEVER delivered messages inside the bot binary despite a passing standalone test, across every variant tried (create()-allocated channel structs, module-level globals, {.threadvar.} globals which wrongly make per-thread copies so the worker writes a different variable than main reads, and atomic-flag-gated blocking recv() which blocks forever) -- a passing standalone tryRecv test does not prove it works in the real binary. Working alternatives: own runtime state PER INSTANCE (module-global LLM channels/worker Thread caused cross-bot event leakage and a shutdown hang where close() joined the shared worker in an 8-agent match; test under the real multi-agent harness; do not pass a pointer into a resizable array to a worker thread -- store per-bot runtime behind a stable heap pointer); make worker state thread-local or pass it through channels; create fresh per-call resources (new HTTP pool per request); have the worker serialize trace events to plain JSON strings pushed over a channel and drained on main; or subprocess polling (spawn process, poll process.running() each frame, kept the frame loop alive 1970+ frames). For a Nim-perception-plus-Python-brain bot define an explicit typed message protocol over a Unix-domain socket using newline-delimited JSON with three types: snapshot (Nim->Python every N frames), directive (Python->Nim: strategy/targets/chat/navigation/hold/until), event (Nim->Python on state changes). Transport NEGATIVE: avoid shelling out to curl/Python per LLM call (higher latency) and avoid a full Python rewrite (numpy per-frame localization risks being too slow); prefer in-process or Unix-socket/pipe/local-HTTP with a Nim fast path and a non-blocking Python slow path. Pass an explicit timeout into the LLM HTTP call, map timeouts distinctly (LlmTimeout vs LlmHttpError), and keep shutdown idempotent.

#### 11. Native FFI worker: init explicitly from a native-owned thread, never lazily on the hot path; audit teardown across the FFI boundary
`generic` · **negative result**

Spawning a native worker thread lazily from inside a per-frame FFI call running on a foreign (Python ctypes) thread races on shared channels/state and produces memory corruption (deadlock, SIGSEGV, or caught-exception noop depending on timing). Start any background worker once via an explicit init entrypoint from a thread the native side controls, never lazily from the hot per-frame path on caller-owned threads. Whether an FFI-loaded library auto-initializes its runtime on load is checkable, not guessable: inspect the compiled dylib for a constructor / mod_init_func section rather than assuming a build flag like --app:lib guarantees one. Lifecycle/teardown that writes analysis artifacts can be silently severed across an FFI boundary: trace manifests stayed 'closed:false' with missing end_tick/outcome/role because there was no destroy_policy export on the native side and no close()/__del__ on the Python wrapper to call it, so closeTrace/writeManifest(final=true) never ran -- when episode artifacts are incomplete, audit the whole teardown call chain end to end, not just the writer. A crash that only surfaces during shutdown can hide the real per-frame failure: a SIGSEGV appeared in teardown (destroyBot during policy.close()) but in the real match SIGTERM killed the process before close() ran, so only the silent noop showed; when a bug reproduces in your harness but not production (or vice versa), check whether teardown order differs. Distinguish 'gameplay broke' from 'shutdown broke': with a fake API key gameplay still moved (~45% non-noop) but the process never exited because policy.close() joined a worker blocked in an in-flight HTTP call.

#### 15. Design match-request schemas as a per-seat discriminated-union model; trace every field end-to-end before claiming it is load-bearing
`loop` · **negative result** · ⚠ _session-derived, unverified_

A roster/matchmaking request grown by accretion (explicit policy_version_ids, then named-player and top-N selectors) ends up with fields meaningful for some modes and ignored for others, and offers two mutually exclusive ways to say 'who am I': a direct caller-owned policy-version list (allows explicit slot assignments but forces the whole roster to be yours) vs a league 'requester' selector (pulls in opponents but forbids explicit assignments), so you cannot mix your bot with others' and control seating in one request. #### 33. Expect scripted policies to plateau ~top 10% against learning competitors; the gap is mining, disruption, and coordination
`loop`

A pure scripted (no-learning, no-LLM-in-hot-path, no cross-episode persistence) policy iterated over 8 ships plateaued at rank 68 of 711 (~top 10%) while the learning-based leaderboard top ten clustered at score 41-42. The ~8-point gap decomposed into cleaner mining (~3), dedicated enemy-disruption agents (~3), and coordinated multi-agent timing (~2).

# Build, package & submit — guide

Reference notes, design rationale, and negative results (25) for build, package & submit. Read when working in this area.

#### 1. Coworld image upload is a multi-step v2 flow; image upload alone registers no role
`loop`

Uploading a Coworld policy image is multi-step, not one endpoint: POST /v2/container_images/upload, push to the scoped ECR target it returns, /upload/complete, then create the Docker-image policy version (image must be linux/amd64). Contract lives in metta app_backend/src/metta/app_backend/v2/routes/experience_requests.py and stats_routes.py. Uploading an image only creates a ContainerImage row and pushes to ECR — it does NOT modify a coworld manifest. To make a NEW role (e.g. reporter) appear you must separately POST /v2/coworlds/upload with a full manifest referencing the img_... IDs; the CLI normally chains both, so calling the low-level image route directly stops at phase 1 and the role never appears live. Do NOT use upload-policy/package-upload to register a reporter image — wrong tool. Note upload-policy pushes layers/manifest via the OCI distribution HTTP API (not `docker push`) to sidestep a Docker 29 + ECR 403 bug, and needs AWS CLI v2 on PATH because it shells out to `aws ecr get-login-password` (no AWS account needed; creds come from the upload pre-sign).

#### 2. Manifests are the source of truth; treat the downloaded one as a local artifact and edit minimally
`loop`

Manifests tie a policy together: coplayer_manifest.json names the policy, the container image, the binary path (/bin/<bot>), and target game; the game manifest (coworld_manifest.json) declares game version and supported league variants (e.g. standard-8p-2i). Adding a variant means a manifest version bump + variant entry, separate from bot source; names can diverge (the internal Coworld adapter short name may differ from the manifest name), so verify both. CoworldManifest and all role specs use Pydantic extra="forbid", so any unknown field in a player[] entry fails validation; a runnable entry has THREE required fields (id unique/referenced by certification, name, type, non-empty description, image — plus optional `run` argv, `env`, source_url; docs in COWORLD_REFERENCE.md and MANIFEST_README.md miscount). When editing manifests make a MINIMAL top-level key insertion preserving formatting (a structured rewrite expanding compact JSON arrays makes the diff unreviewable) and validate before commit: every coworld_manifest*.json parses, grader lists are non-empty where required, `git diff --check` passes (watch a doubled space after a colon). The resolved-manifest schema model can falsely reject a SOURCE manifest using relative protocol-doc paths — validate source manifests with a raw JSON/schema check. Coworld uploads store documentation as PUBLIC URLs and do NOT bundle local Markdown, so updating local docs alone does not change what an uploaded policy/game exposes.

#### 3. Current coworld manifest schema rules: equal token min/max, no per-slot names, digest-pinned image
`loop` · **negative result** · ⚠ _session-derived, unverified_

Read the live schema metadata (x-coworld-future-required) rather than session memory to settle which manifest roles are required: a coworld has seven roles (game, player, commissioner, reporter, grader, diagnoser, optimizer) but sources disagree on whether only game+player are required. Concrete validator rules: game.config_schema.properties.tokens must declare EQUAL minItems and maxItems (a range like 0..16 breaks infer_fixed_token_count); per-slot names are GONE (both slots.items.properties.name and top-level player_names are rejected, landed metta PR #14678) so display names go in game_config.players[].name. Pin the manifest image_uri to an immutable digest (image@sha256:...) rather than 'latest' so the runner gets exactly the image you validated. The player[].env block fully REPLACES the player's public environment (anything not declared falls back to the Dockerfile ENV), so a declared manifest env var overrides the baked-in default at deploy time — use env only for values selected per deployment. The worked-example pointer is worlds/paintarena/coworld_manifest_template.json.

#### 4. A coworld player is just a linux/amd64 image speaking the protocol; ship per-player as a sibling, build hermetically
`loop` · **negative result**

Treat a coworld player as nothing more than a linux/amd64 Docker image speaking the game protocol over a websocket; upload-policy and manifest entries sit on top of that single contract. Design players around 'produce a conformant image', not 'an importable Python package', so each is decoupled and shippable on its own; the package boundary should match the dependency boundary, not a conceptual grouping. Do NOT nest a new player inside an existing one with heavy eager imports (nesting coborg_among_them inside scripted among_them/ pulled the parent's eager mettagrid import into the new container, bloating it) — make it a sibling; organize concrete policies as standalone projects each shipping via its OWN build.sh (shared utils like cogsguard's are the limited exception). Build hermetically with Docker as the only host requirement and no credentials when inputs are public: fetch the Nim toolchain, game repo, Python packages, and shared SDK inside the image build, each pinned to a ref; keep the image lean (python:3.12-slim, install only real runtime deps so they cache independently of source, then `pip install --no-deps` for the package, omit the heavy cogames extra which pulls a Bazel+Nim build). One image feeds two INDEPENDENT shipping paths: an uploaded policy version (upload-policy, league-submittable) and a bundled player[] entry in a game author's coworld_manifest.json (via upload-coworld); the same image can serve either or both, so don't conflate 'submit to a league' with 'bundle into a coworld package'.

#### 5. Standardize a per-player build.sh: three artifacts, CMD not ENTRYPOINT, rebuild from fresh source, dated pin, one tag
`loop` · **negative result** · tool: `build.sh`

A Coworld player ships as a self-contained Docker container whose build.sh produces three artifacts: a linux/amd64 image (deterministically locally-tagged, no mandatory registry push; --tag/--push <ref> override), a coworld_manifest.json player[] snippet emitted to BOTH stdout and a --manifest-out/-o file, and a documentary dist/coplayer_manifest.json sidecar ({author,name,image_uri,games}) that upload-policy/submit/certify/upload-coworld/run-episode ALL IGNORE (purely documentary). build.sh must run from any cwd, support --no-build (render manifests only), and REBUILD the image + regenerate the manifest from the freshest code in the player dir (never point at a pre-existing image), so a top-level walker can `find players -name build.sh -exec {} \;` with no orchestration. Build with CMD not ENTRYPOINT so the runner executes the manifest binary with appended runtime args; smoke locally by running it with no args and with --help, and run every build.sh for real against a live Docker daemon (standardizing files and verifying they execute are separate steps). Pin the workspace against the source-of-truth packaging contract with a DATE ('pinned against Coworld source as of <date>') so drift is visible. Use ONE image-tag convention across build script, run script, Dockerfile, and manifest: play_local.sh defaulting to a different (underscored) tag than build.sh produced (hyphenated coborg-among-them:dev) silently built a second parallel image.

#### 6. cogames CLI/server code map and submission pipeline (ship vs the four sub-commands, with -f)
`loop` · tool: `cogames`

The cogames entry point is cogames.main:app (Typer); ALL commands live in one ~2500-line packages/cogames/src/cogames/main.py. Bundle creation, Docker validation, and S3 upload are in cogames/cli/submit.py; the tournament-backend HTTP client is cli/client.py (TournamentServerClient); cli/generated_models.py is auto-generated from openapi.json (regenerate, don't hand-edit). The tournament SERVER is NOT in cogames: it is the FastAPI/SQLModel app_backend (submit endpoints routes/stats_routes.py /stats/policies/submit/*, tournament routes/tournament_routes.py). Auth subcommands wrap softmax-cli/softmax-common token storage. Custom forks derive from cogames/policy/starter_agent.py (StarterPolicy) — diff against it. The pipeline is four commands: create-bundle -> validate-bundle (Docker smoke) -> upload (bundle+validate+upload, optionally submit) -> submit (attach uploaded bundle to a season); `cogames ship` chains all four (canonical `cogames ship -p class=<module.Class> -n <league-name> -f <package-folder> --season <season>`), drop to lower-level commands to retry one stage. For a local (non-installed) policy you MUST pass the policy AND every local module it imports with -f — the bundle flattens its include set into one root and the container rebuilds the shared library from source, so all transitive deps (sibling modules, common/, engine source tree, baked navigation/assets, locks like nimby.lock) must be reachable inside that root; verify the import closure (grep / `ship --dry-run`) rather than assuming. `--dry-run` validation runs locally inside the tournament Docker image and only catches crashes and ABI/build errors, NOT bad behavior. Diagnose async failures with `cogames matches <id> --logs` and `cogames match-artifacts <id> logs|error-info`.

`loop` · **negative result** · ⚠ _session-derived, unverified_

#### 8. Reporter output: single deterministic zip with in-zip render manifest, inline CSS only
`loop` · **negative result** · ⚠ _session-derived, unverified_

For coworld reporter output prefer one .zip over a JSON envelope: the envelope inflates binary ~33% via base64, needs a content_type-discriminated polymorphic field, and forces artifact-id Markdown substitution. A zip makes each artifact a real file with native bytes and normal filename links; infer content-type from extension via mimetypes. Flag the render target with an in-zip manifest.json (reporter_id/render/event_log; canonical) or render.txt of filenames (pre-canonical), allowing only .md/.html (.htm is non-canonical). For byte-identical zips on identical input, pin every entry mtime to (1980,1,1,0,0,0) AND pin pyarrow. Reporter HTML renders in Observatory's iframe/CSP sandbox so use inline CSS only (no <script>/<link>/external fetch); XSS, sandboxing, and zip-safety are the consumer's responsibility. When the reporter interface is undefined, assume the simplest schema-meeting contract (handed a bundle zip to unpack) rather than building URI fetching. Migrating a pre-canonical reporter changes only the I/O envelope and Parquet dtypes — artifacts (summary.html, stats.json, events/proximity.parquet) carry over; pre-canonical takes five input env vars (COGAME_RESULTS_URI, COGAME_REPLAY_URI, COGAME_EPISODE_METADATA_URI, COGAME_LOG_URI, COGAME_REPORTER_ID), writes COGAME_REPORT_OUTPUT_URI, flags render with render.txt; canonical collapses input to one COGAME_EPISODE_BUNDLE_URI (a bundle zip with inner manifest.json), writes COGAME_REPORT_URI, uses an in-zip manifest.json. If adding a manifest reporter entry would break an older validator AND the runtime that invokes reporters has not shipped, defer the manifest change and document the intended entry in the README. When replacing the default reporter, keep ONLY yours in reporter[]. Reporter image bloat comes from pyarrow (~139MB of ~191MB site-packages) and the Python base, not the crewrift tree (~13MB); target pyarrow first and gate Parquet behind an optional extra (players[trace-parquet]) so JSON/JSONL/CSV stay stdlib-always-on.

#### 9. Bake static derived data offline into a vendored, self-contained image; never compute it on the first tick
`crewrift` · **negative result**

A hosted Coworld policy image must be fully self-contained — it cannot assume repo files at runtime and the runner does NOT auto-provide reference bots or assets; vendor everything the bot needs (map geometry, sprites, reference code, font glyphs). Bake static derived data (a pure function of static inputs) into the image OFFLINE, never per-run: crewborg's nav graph + occupancy substrate are a pure function of the one static croatoan map, but building them in pure Python on the first tick cost ~13.7-14s (up to ~29s) under the hosted 250m-CPU cap, freezing the agent at spawn while the 24 Hz engine streamed ~330 frames ahead (the 'slow to leave start' symptom). Fix: bake once into a vendored asset (croatoan_navbake.pkl.gz) loaded at runtime (~0.1s, tick-1 step 52-72ms vs ~13,700ms, ~200-280x) with byte-identical play; validate the streamed walkability mask against the bake and fall back to a live build on mismatch/load error so correctness never depends on the asset, and on mismatch conclude the server runs a different map and fail loud. Do not make an offline generator a hard build dependency: check in the generated artifact, package it via existing data globs, and keep the engine/toolchain out of the Docker build (regeneration stays an explicit occasional step); let the bake tool take heavier deps than the runtime bot. When porting to a new language do NOT reimplement asset/map loaders (e.g. Aseprite parsing): pre-render once with the existing (Nim) stack via an offline bake tool emitting compact artifacts the new (Python) policy loads with numpy/PIL; when a reference parser's core helper is private, copy the small (~30-line) helper inline (documented as a replica) over importing the whole module. Bake provenance (source engine commit hash in artifact metadata), never an absolute /Users/... path.

#### 10. Crewrift static map facts live only in the vendored resource file; never arrive at runtime
`crewrift`

Crewrift vent, emergency-button, and task locations are NOT in the stream (the map object is a flat prerendered picture); they live only in the server-side map resource file (data/croatoan.resources, baked into the server image) that is never delivered to a player. A bot author without game-repo access cannot obtain them over the protocol, so vendor and parse the resource file at BUILD time, parsing the static map into belief at container startup (never per tick — the map is static for an episode). The file-order of task rects IS the 3000/7000+idx stream index, and the emergency button is a derived 28x34 rect centered on the bridge room. Building private starter crewrift policies (notsus, suspectra) needs a Metta-AI-org PAT (private repos); gate the build with informative errors and clean up token plumbing once the repos go public.

#### 11. Adding a new Crewrift player: template, context, nothing else
`crewrift` · tool: `build_player.sh`

Adding a new Crewrift player needs only three things wired into the build: a Dockerfile following the Nim template (clone public game at ARG CREWRIFT_REF, nimby --global sync, overlay source, nim c, slim runtime stage) OR the Python template (pip install deps + SDK, package on PYTHONPATH, bake CMD python -m <module>); a context decision (self-contained dir vs needs the shared SDK from the pinned public players repo); and nothing else if the player is not vendored — vendoring only changes where source is copied from, not how the image is built. The smallest complete from-scratch baseline is the Nim 'notsus' reference (decode Sprite-v1, move, press A), which doubles as a comparison opponent; crewborg (Python) is the full worked player and the source-verified reference for every id-range/label/offset; suspectra is the reference for wiring a latency-bounded meeting LLM. crewborg pip-installs player_sdk from the public players repo tarball at the pinned ref + copies the staged fork onto PYTHONPATH (deliberately NOT pip installing the lab package, which would pull coworld[auth]); notsus/suspectra clone the crewrift game at pinned CREWRIFT_REF and compile (a Nim-toolchain image incurs a multi-minute cold build). #### 12. Give bundled players an exit-on-disconnect mode for certification; keep the upload template results schema in sync
`loop` · **negative result** · ⚠ _session-derived, unverified_

Coworld certification runs the game plus its bundled player containers and waits ~10s for those containers to EXIT; a bundled player that reconnects forever on WebSocket disconnect fails with 'Timed out waiting for player container to exit'. Give the bundled player an explicit exit-on-disconnect mode (e.g. `notsus --exit-on-disconnect`) so it exits cleanly when the game closes the connection during supervised certification, while keeping reconnect-by-default for live play. Placing the flag in the manifest's top-level player `run` array (e.g. ["/bin/notsus","--exit-on-disconnect"]) affects EVERY use of the bundled player (certification, local play, smoke, starter examples) but NOT externally-submitted league containers — treat it as a property of the bundled runnable. Separately, the upload template's results schema can drift from the game's actually-emitted results and silently block certification: after fixing a player-exit issue, certification failed again because the upload manifest template rejected connect_timeout/disconnect_timeout result fields the game's own coworld_manifest.json already emitted — when the game adds result fields, update the upload template's results schema to allow them.

#### 13. The Coworld seed reconciler picks the latest manifest by created_at (shared-league footgun)
`loop` · **negative result** · ⚠ _session-derived, unverified_

The Coworld seed reconciler is a shared-league footgun: it selects the latest coworlds row by created_at (not by owner or semantic version) and rewrites the daily league's default env config from that manifest's variants. Any account uploading a newer Coworld whose manifest exposes only a smoke variant silently reverts the league to the smoke config (confirmed live when another account's smoke-only upload reverted the standard 8p2i default within minutes). Fix the class of bug durably: patch the canonical/team-source manifest to include the real league variant (e.g. standard-8p-2i) so re-seeding can't drop it, and/or change the reconciler to exclude certification-only variants from league defaults; a live one-off re-upload only holds until the next conflicting upload. Keep the cheap certification smoke config under 'certification' only and expose just the real gameplay variant in the manifest's public 'variants' list so it cannot become a scored daily round.

#### 14. Use replay metadata to confirm what config a match actually ran
`crewrift` · ⚠ _session-derived, unverified_

Replay metadata reliably confirms what config a cogames tournament match ran — use it instead of assuming a hidden 'competition mode'. #### 15. Stage cogames-to-coworld migration as dependency-ordered phases; confirm per-player scope and audit old-CLI references
`loop` · **negative result**

Structure a coworld migration as explicit phases: Phase 0 Discovery captures the real cow_/league_/div_/variant IDs plus the mettagrid pin from the live registry (it blocks every later phase), then dependency swap, local `coworld download` + `coworld play` validation against the BUNDLED starter player (failures isolate before your image enters the loop), Dockerfile swap, first submission via upload-policy/submit, documentation rewrite LAST (it bakes in real league/division IDs). Concentrate work in three files: pyproject.toml (dependencies), Dockerfile (build/install), and the tournament-submission section of the docs; runtime policy code importing only mettagrid.* is unaffected. Do NOT assume every player migrates (one project froze alpha_cog on deprecated cogames, migrated only bulbacog) and record deprecations in durable memory; do not carry forward old league/division/variant names. Actively migrate policies off a deprecated CLI rather than assuming compatibility (old CLIs silently emit wrong deployment shapes, e.g. a generated run_agent.py WebSocket shape and docs path both wrong) and audit EVERY place the pipeline references the old CLI, not just the entrypoint. Scaffolded Coworld Dockerfiles may COPY viewer/client assets that don't exist at the pinned commit — a headless build doesn't need them, so delete the offending COPY line rather than fabricating the path. Recover the bundled reference player by cat-ing it out of the downloaded game image with a docker entrypoint override. Before touching the player, delete dead scaffolding the plan would force you to rewrite, but first verify zero inbound imports (removing a ~1492-LoC railcar TUI debugger plus two unreferenced design docs deleted a whole section of doc-rewrite work). Validate the new toolchain against the official starter player before changing your own image so a later failure is attributable to your image, not the toolchain.

#### 17. Vendor the player as a fork but import the shared SDK editable; a copied entrypoint is a soft fork to track

#### 18. Player packaging layout drifts across game generations; prefer the modern foldered shape and preserve name->policy resolution
`loop` · ⚠ _session-derived, unverified_

A coworld player's packaging layout drifts across game generations: older players used a top-level entry plus a manifest-only folder; the current convention is a self-contained per-player folder (players/<name>/) holding source, a coplayer_manifest.json, a build script, and a cogames/ subpackage. Prefer the modern foldered layout for new clones, and before copying a build/packaging file confirm it still exists (git history may show it was deleted in a 'players in their own folders' refactor) and verify behavior against the actual CLI implementation rather than prose docs that lagged the reorg.

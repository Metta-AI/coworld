# Navigation & pathfinding — guide

Reference notes, design rationale, and negative results (36) for navigation & pathfinding. Read when working in this area.

#### 1. Fix wedging at the source (clearance + replan), not with a stuck-detector
`crewrift` · **negative result**

A stuck-detector is a bandaid, not a root-cause fix, and aggressive stuck-recovery can hurt score (one crewrift loop reverted it; clearance-aware pathing dropped the worst wedge from 1167 ticks permanent to ~96 ticks self-resolving). Crewborg root cause: plan_route string-pulls segments by pixel line-of-sight so a segment counts clear if it merely grazes a corner (zero clearance), and the bang-bang axis-aligned d-pad follower staircases off any non-axis line, so lateral drift on a wall-hugging segment hits the wall, and because it only replanned on goal change it pressed into the wall forever (the walkability mask was correct, so this is a control+clearance problem, not a mask/server mismatch). Two fixes: (A) clearance-aware pathing -- build an ERODED mask where a pixel is 'clear' iff its (2r+1)^2 box is fully walkable (expand walls by the player half-width for footprint fit, e.g. 7x7), place graph nodes and string-pull routes down corridor CENTERS using that eroded mask, but keep edges and the reachability flood on the TRUE un-eroded mask so tight passages and wall-adjacent destinations stay reachable; (B) periodic re-planning for static-goal intents every 4-10 ticks. Diagnose with the held-button mask in traces: mask 32 = holding A (doing a task) but a constant diagonal like 9 (UP+RIGHT) for hundreds of frames = pressing into a wall (the viewer's PlayerN labels do NOT map to your connection slots, so scan all slots).

#### 2. Stuck-recovery firing is not working; invest in avoidance and cross-check stats
`loop`

Stuck-recovery FIRING is not the same as WORKING: an agent spending 26% of an episode stuck despite a 120-tick detection window means recovery fired but never escaped, typically because it is trapped in an enemy-territory pocket where every reachable target is gated behind enemy cells, so the repick keeps choosing unreachable goals. Stuck AVOIDANCE beats stuck RECOVERY -- top competitive players basically never get stuck (best tournament max_stuck ~14 vs our agents at 2500-5875), so invest in not entering traps. Recovery refinement is also high-value only up to a point: every refinement delivered a measurable self-play A/B win (5+ score total), but once recovery is good enough the agent's dominant failure mode shifts from being-stuck to being-KILLED and further stuck work delivers ~half its predicted value, so re-check which failure mode currently dominates (replay + stuck-tick metrics) before investing more in an already-saturated mechanism. Conversely, a high max_steps_without_motion (e.g. 3529, 35% of episode) under A* navigation does NOT necessarily mean stuck-detection is broken: it can mean agents are dead/respawning or wall-impeded in unfamiliar enemy territory, so cross-check against the move-failure rate (e.g. 19%, ~1931 failed moves/agent) before concluding recovery failed.

#### 3. Landmark-first localization; cross-validate before writing the map
`crewrift` · **negative result**

Landmarking is the primary absolute-localization method; dead reckoning is only per-tick movement tracking between landmark sightings. Localize an egocentric (radius-6, 13x13 window) agent by scanning the observation for objects whose absolute position is known a priori: if a known landmark appears at egocentric offset (er,ec), absolute pos = known_pos - (er-6, ec-6); since agents spawn adjacent to fixed structure, localization should resolve on tick 1 -- do NOT pixel-match (in Coworld you render at screen center, so the map object's screen position alone gives world coordinates). For screen-readers that DO match pixels, search LOCALLY near the previous camera first (seeded from a known pre-game point) and spiral only from a good seed; a naive full-map top-left scan every frame was the worst CPU problem (hundreds of ms when seeded wrong). Fixed compound landmarks (hub, stations, compound extractors) are the source of truth and win when visible: if they imply a different position, spatial memory corrects to the landmark-derived position BEFORE writing observed cells into the global map, or an unvalidated/dead-reckon estimate smears a frame of bad map data (shifted landmark observations written as phantom impassable objects that block A* and produce repeated 'noop, no path' warnings on a navigable map). Treat hub/stations as hard anchors; allow a single safe landmark to correct during startup but require multiple agreeing landmarks (or a hub/station) for later corrections. Do NOT add procedural/seed-specific junctions as landmarks. Dead reckoning must use the engine-APPLIED action (decoded from the global last_action / lp:* displacement token), not the action the policy emitted (reusing the emitted _last_action caused drift); prefer drift-free ground-truth displacement over self-integrated position when available. NEGATIVE RESULT: zero-valued displacement tokens are omitted at spawn, so a spawn-relative anchor never initializes from spawn alone -- a partial fix adding only spawn-anchor logic regressed a full run to 0.0 because the anchor must ALSO refresh when a later landmark observation agrees with current position ('anchor learned late' is a distinct failure from 'anchor never learned').

#### 4. Rank repair candidates by local reachability; leash HP at the mode level
`crewrift` · **negative result** · ⚠ _session-derived, unverified_

Rank frontier repair/rebuild target candidates by LOCAL reachability, not global graph importance: repeated graph-priority attempts regressed because they pulled agents toward globally high-value but locally expensive bridge targets (sometimes effectively inside Clips space), so order candidates by travel distance first, then local chain reconnection/support, deliberately NOT by global bridge value, preferring nearby capturable neutral junctions that reconnect a local chain on the friendly side. NEGATIVE RESULT: safer-routing and HP-survivability mechanisms repeatedly fail to demonstrate value, especially when the pathfinder already avoids the danger they guard against. Safer-routing fixes that veto leaving friendly territory cut productive expansion harder than they cut deaths (broad route-HP safety collapsed seed 42 to 4.77; clean Clips-capture safety helped 42/99 but collapsed seed 7 to 12.73; scrambler heart-preservation dropped seed 42 to 15.34). A miner HP cargo gate (deposit early if HP critically low to avoid losing a 40-unit cargo on death) showed no value in self-play: it fired too broadly in neutral territory (-0.64 at 10k) or too narrowly in enemy cells, which miners rarely occupy because V15's A* danger_cost=3 already routes them around enemy territory, making the gate inert/byte-identical; depositing at a nearby friendly junction instead of the hub was likewise rejected (0W/11T/1L, -0.066). Prefer mode-level HP leashes over target-level HP vetoes. Aligners should escort a raiding scrambler only to a safe frontier anchor near friendly territory and wait, never following it into enemy territory toward a deep unconnectable corner; the scrambler must prefer enemy junctions that project back into friendly-held frontier. When selecting safety/retreat targets, exclude the current cell whenever the live territory:here sensor says the agent is not on friendly ground (trust the live sensor over inferred memory for the agent's own cell), or the retreat targets the current cell and stalls.

#### 5. Model the cogs-vs-clips energy/HP economy; energy-failed moves look like walls
`crewrift`

Energy and HP are active mechanics a nav policy must respect. Movement costs 4 energy/step; a day/night cycle (day_length 200) grants solar +3/tick day, +1/night; FRIENDLY territory regen (+100/tick plus solar) far exceeds the base energy cap of 20, so agents in territory are always full and move freely, while OUTSIDE territory they move only ~every 1-2 ticks by day and ~every 4 ticks at night. HP: base 100 (initial 50 in cogsguard), -1/tick always plus an extra -1/tick in enemy territory, healed to full in friendly territory; at HP 0 hearts and gear are cleared. avoid_enemy_territory_radius filters only TARGET selection, not in-transit pathing, so a long mission can die mid-transit and silently lose gear. CRITICAL: an energy-stalled failed move is INDISTINGUISHABLE from a wall/agent block in the move token (the sim executes noop), so a policy that ignores energy will misread exhaustion as obstruction and thrash. Rather than building heavy 'go recharge' logic, detect the failure (intended move != executed action) and RETRY up to a small cap (default 3), since agents auto-regen each tick and regen fully near aligned buildings. Trigger retreat on distance-aware vitals (should_retreat if hp<30 OR energy<8 OR hp < anchor_distance+15) and only energy needs a rest ceiling since friendly HP regen is effectively instant. Energy-aware behavior is a large untapped navigation lever: prefer targets in or near friendly territory at night, noop to REST when energy is low instead of burning failed-move ticks, and extend territory to create energy-healing corridors.

#### 6. Bake navigation data offline from the authoritative geometry source
`crewrift` · ⚠ _session-derived, unverified_

Building a nav graph on tick 1 (flood-fill plus per-pixel node/edge over an ~814k-pixel walkability mask plus an O(anchors^2) A* substrate) cost ~14s on a 1/4-core prod host, stalling the first frame and blowing the per-tick budget. Bake MapData + the full NavGraph once into a checked-in serialized asset (nav_graph.json nodes id/x/y/kind/room/label plus edges; an offline A* baker walks the walk_mask and emits nav_paths.bin; ship as .npz in the player image), load at startup, and treat the streamed walkability mask as a dimension/contract check, not a build input. Bake from the SAME walkability source the server uses (semantic resource rectangles are NOT the pixel-level art-layer mask; the wrong source silently produces a graph that disagrees with the live sim); validate the streamed mask against the asset at runtime and fall back to a live build on mismatch so correctness never depends on the asset.

#### 7. Spend bake budget on pixel-correct reachability and interaction anchors
`crewrift`

Crewrift collides the player as a 1x1 point, so every walkable pixel is a legal position and reachability is one connected component flood-filled from spawn (scipy flood costs ~0.6-1.2s worst case on a fully-open ~813k-pixel mask; run it during camera/stream warmup so it is a one-time startup stall, not per-tick). A coarse (8px) nav graph for A* speed is fine ONLY if correctness stays at pixel resolution: a cell is a node iff it contains a reachable walkable pixel (place the node ON that pixel, not the cell center), join 8-neighbor nodes only when the connecting pixel segment is fully walkable (no diagonal corner squeeze). For each task/vent/button compute a baked anchor = the reachable pixel satisfying the interaction condition (inside task rect / within VentRange of vent center / inside button rect) nearest the natural center, and navigate to that anchor, never the raw rect center.

#### 8. Verify a baked asset is actually decoded and non-empty at runtime
`loop` · **negative result** · ⚠ _session-derived, unverified_

A precomputed asset is only 'wired in' if a runtime parser actually decodes it; embedding the blob is not enough. A baked nav-paths binary was embedded but the loader set paths to an empty list and never decoded it, leaving the path-following code dead while the bot ran on a straight-line fallback; separately, nav graph blobs set to empty strings under a 'TEMP disabled' comment made nearest-waypoint distance effectively infinite and the policy emitted an empty action mask every tick (a statue). When a refactor splits produce-asset from consume-asset, assert the consumed collection is non-empty at startup so degenerate nav/perception data fails loudly. Watch for the related smell: fields read or initialized 'for X' but never assigned (lastSelfX/lastSelfY for unimplemented drift detection; a ventPolicy field never set to anything but VentNever, silently disabling vent routing) signal an incomplete refactor; grep for them.

#### 9. Use the right navigator per target regime and audit each independently
`loop` · ⚠ _session-derived, unverified_

A policy with multiple navigators computes each goal independently and they can diverge, so audit each separately and reproduce a bug against the regime that triggers it. In alpha_cog pursue_target, A* is used ONLY for REMEMBERED (out-of-view) targets (gated by args.astar_navigation); VISIBLE targets use BFS within the 13x13 ego window (bfs_to_visible_target) or direct adjacency, which is sufficient and cheaper -- do not unify the two blindly. In the transit-through-interactive bug the BFS and greedy navigators correctly used the target's FREE NEIGHBORS as goal; only the A* navigator pathed to the target cell itself and was buggy, so the bug lived in only the remembered/out-of-view regime.

#### 10. Lookahead steering must be wall-aware and direction-correct
`loop` · **negative result** · ⚠ _session-derived, unverified_

NEGATIVE RESULT: pure-pursuit/lookahead steering (advance N pixels along the baked path, steer straight at that point) drives the agent into walls when the path bends sharply, because the lookahead target lands on the far side of a wall; even bake-time simplification does not guarantee the straight line between two retained waypoints is traversable. Lookahead distance is a tradeoff (smaller is safer near tight geometry, sluggish in open space), so prefer ADAPTIVE lookahead or wall-aware target selection over one global constant. Separately, a monotonic-only progress variable (pathProgress = max(pathProgress, snapIdx)) is a trap on REVERSIBLE edges: if the path is stored in fixed source->destination order, traversing it in reverse snaps progress near the far end and pins the bot toward the wrong end -- directional path data needs a direction parameter and reverse-indexing, not just clamping progress upward.
  <sub>sources: opencode:ses_1fb1d9989ffeYXiew23WUqo2hw, opencode:ses_201972733ffexPIP9mq6yEDpqJ</sub>

#### 11. Prefer target-selection avoidance over per-tick nav avoidance (except full-plan A*)
`loop` · **negative result**

NEGATIVE RESULT: prefer target-selection-level avoidance (changing WHERE agents go) over per-tick navigation-level avoidance (changing HOW they get there) for danger/territory -- filtering targets in enemy territory shipped a horizon-scaling win, whereas per-tick territory-aware greedy nav and full-4-direction BFS both regressed at 10k with huge per-seed variance (E035 full-BFS: +0.124 at 2k but -0.315 at 10k; +5.54 on seed 256, -11.76 on seed 7). Root cause: changing even one agent's intermediate path shifts which cells teammates observe, cascading position changes that break the aligner cycle, and the biased-BFS failure mode is load-bearing for downstream flags. Gate any per-tick nav change to confirmed no-progress episodes. CRITICAL CAVEAT: target-selection avoidance does NOT prevent A* routing THROUGH enemy cells; alpha_cog's avoid_enemy_territory_radius filters only selection, so once A* plots through enemy territory the agent walks into the -1 HP/tick zone -- in-transit safety needs a separate finite-A*-penalty mechanism (which is the one per-tick exception that composes, since A* commits a full plan).

#### 12. Death loops: model damage as geometry, and blacklisting lethal cells fails
`loop` · **negative result** · ⚠ _session-derived, unverified_

In cogs-vs-clips, deaths are a primary score driver: at HP=0 an agent loses all carried gear AND hearts (~34 elements per death), so 30 deaths can destroy ~1020 elements vs ~412 ever deposited -- deaths can destroy more value than mining produces. Damage is from PROXIMITY (~-1 HP/tick within radius 10 of an enemy junction, radius 20 of an enemy hub), so an aligner in enemy territory survives only ~100-150 ticks; treat survival as a positional/geometry problem (route around enemy structures, retreat when HP low), not combat. Watch for the death SPIRAL: a structural positive-feedback failure with no circuit breaker (more enemy territory -> more deaths in HP zones -> less mining/hearts/alignments -> further enemy expansion); 44% of tournament deaths were one agent repeatedly targeting the same lethal junction, dying, re-gearing, and walking straight back (one died 6 times at the same cell in 3000 ticks). NEGATIVE RESULT: the obvious fix -- a position-based death-target blacklist excluding recently-lethal cells -- was rejected in every config; under stressed 6+2 + angry_clips ALL frontier junctions are lethal so blacklisting just sends the agent to an equally-lethal alternative, and in 8+0 self-play it suppresses normal one-off productive deaths. Its core assumption that a safe alternative exists fails exactly when death loops matter; the real fix needs path-cost/territory-aware filtering distinguishing 'permanently unreachable' from 'occasionally lethal but still best'.

#### 13. Before fixing a 'missed target', distinguish out-of-range from ignored-in-range
`crewrift` · ⚠ _session-derived, unverified_

Before treating a passed-over neutral junction as a targeting bug in cogs-vs-clips, check distance to the nearest Cogs network: many 'missed' neutrals are simply not capturable yet. Replay showed agents passing neutrals 16-42 tiles from the nearest Cogs network (out of capture range); the real bug was an aligner locking onto a far directional target and walking past a cheap in-range local capture. Distinguish 'out of range' from 'ignored in range' before coding a fix. Treat a spatial-memory cell-classification bug (junctions stored/classified as plain floors) as a deep root-cause investigation, not a symptom patch -- misclassifying cells corrupts pathing, scoring targets, and likely other cell types -- and demand a root-cause report before reclassifying on the surface.

#### 14. Localize sparingly, never re-validate per tick, and treat a wrong map as worse than none
`loop` · **negative result** · ⚠ _session-derived, unverified_

Localize sparingly: only on the first tick and after death, using dead-reckoning between those events; re-localizing every tick yields wildly wrong positions once the agent leaves spawn. NEGATIVE RESULT: continuous per-step localization cross-validation inside the map update() path itself corrupts the map -- remove it and expose an explicit reset_localization() that clears position/localized flag on a declarative trigger (e.g. agent-death detection) rather than re-validating every tick. Treat a wrong map as worse than no map (all downstream decisions build on it): prefer leaving cells unknown over guessing them, and known-static cells (walls, stations, extractors from direct observation or the fixed spawn layout) must NEVER be overwritten by later 'empty' observations. Position accuracy is load-bearing -- A* starts from spatial.position, so a wrong position yields a path from a ghost cell with a wrong first step.

#### 15. Cogs-vs-Clips has no death/respawn; preserve position across death
`crewrift` · ⚠ _session-derived, unverified_

In cogs-vs-clips (cogsguard/machina_1) there is NO traditional death or respawn: when HP reaches 0 the hp_death handler (cogsguard/game/damage.py:47-56) only runs ClearInventoryMutation on gear and heart -- it does NOT relocate, teleport, freeze, or remove the agent, which stays at its exact cell and keeps moving/acting, just de-equipped. So do NOT write death-detection or respawn-recovery logic expecting a position jump, and a purely incremental position tracker that never resets is acceptable (alpha_cog's estimate starts at an assumed origin, advances only on verified successful moves, marks walls on failed moves, with no death/HP/position-discontinuity detection). Concretely, record the death from the observation (HP->0, gear/hearts cleared, death_count++) but do NOT reset spatial localization in Memory.update(): resetting on death stranded a dead/no-gear agent ~9582 ticks from the compound, whereas preserving position raised seed-42 reward 2.7414 -> ~3.002 and cogs junction gains 19 -> 30.

#### 17. Cogs-vs-Clips substrate quirks: ambient damage, hub impassable, extractor-to-floor
`crewrift` · ⚠ _session-derived, unverified_

In the cogs-vs-clips (machina_1) substrate, territory damage is AMBIENT, not per-cell-tagged: cells within radius ~10 of a Clips junction deal damage but carry no tag, so avoid-enemy-territory logic must model the RADIUS, not look for a cell tag. The hub cell is intentionally impassable -- do not model it as walkable; a test asserting the hub is passable is the test being wrong, not a behavior bug. An exhausted extractor changes from obstacle to FLOOR (the C++ extractor has remove_source_when_empty, so when empty it is removed from the grid and becomes passable), so spatial memory must mark an exhausted-extractor cell WALKABLE and update the A* blocked set. Two load-bearing pathing behaviors any direction-bias change must work around: greedy_toward_remembered treats every tagged cell (including agents) as blocked, and explore_action's prefer_visited=False outer loop dominates direction ranking, so a direction-bias system's preference never fires unless it accounts for that loop.

#### 18. Cogs-vs-Clips economy: scale roles to agents controlled, score compounds early
`crewrift` · ⚠ _session-derived, unverified_

Make role allocation reallocatable in-game and scale it to the agents actually controlled (tournament play is 2-6, never the full 8): treat fixed role assignments as initial only (hard-coding an 8-agent split breaks for fewer agents) and plan mid-game switches (convert a miner to an aligner once resources are plentiful). #### 19. Diagnose a wedge before fixing; test the metric the follower actually uses
`crewrift` · ⚠ _session-derived, unverified_

A crewmate standing still at a task until killed is almost always a navigation wedge, not a completion bug -- distinguish via the held-button mask in traces (mask 32 = holding A, actually doing a task; a constant diagonal like 9 = pressing into a wall), and remember the viewer's PlayerN labels do NOT map to your connection slots so scan all slots. Tests that assert on 'path length' can silently measure the wrong thing: counting simplified-path vertices is not equivalent to the follower's pixel-distance progress along segments, so when you change a lookahead or path constant, re-check that the test selects its fixture by actual polyline pixel length or it passes/fails for the wrong reason. #### 20. A subset-shortcut in front of a full search returns byte-identical results
`generic` · **negative result**

A biased BFS that only tries toward-target directions and falls back to a full BFS produces byte-identical output to always running the full BFS: fallback-on-failure equals full replacement when the primary method is a strict SUBSET of the fallback, because there is no path the biased method finds that the full method would not find first. To get a different result you must change the search itself, not add a subset-shortcut in front of it.
  <sub>sources: alpha_cog/AGENTS.md</sub>

#### 21. A recovery trigger must be able to observe the actual failure state
`generic` · **negative result** · ⚠ _session-derived, unverified_

A stuck/stall detector must not be gated on a precondition the bug itself makes impossible: a detector requiring 'path.len > 0 AND lastEmittedMask != 0' can never fire during an empty-path/noop failure that produces exactly path.len==0 and mask==0. #### 22. Suspect one motion-layer root cause for paired symptoms; never overload a tuning constant
`loop` · ⚠ _session-derived, unverified_

A single broken velocity/motion estimate can produce two distinct symptoms at once: orbiting/overshooting targets (no velocity-based hysteresis) AND sticking on wall corners (the stuck-detector never sees motion change). When both appear together, suspect ONE root cause in the motion/velocity layer rather than two bugs. Relatedly, do not overload one tuning constant: a single CLOSE_DISTANCE used as both the movement arrival deadband and the on-top-of-target interact-proximity check cannot satisfy both -- movement needs a LOOSER deadband (~16-20px) to survive momentum while interaction needs to stay TIGHT (~12px). #### 23. Dead-reckon position from the engine's EXECUTED action, never second-guess success
`crewrift` · **negative result**

Maintain position by dead-reckoning from the engine-reported APPLIED action token (last_action_executed plus its move flag), NOT the action the policy emitted, because the engine may reject or alter it. Position-tracking drift is commonly caused by SECOND-GUESSING last_action: do NOT re-derive whether a move succeeded -- if last_action_executed reports move_X, update unconditionally because the simulator already validated it; re-deciding diverges your estimate from the authoritative result and accumulates drift. Scripted policies often track RELATIVE to their own origin (e.g. centered (100,100)) while the sim uses ABSOLUTE coordinates, so when diagnosing coordinate bugs compare DELTAS, never raw values.

#### 24. Diagnose movement bugs with the action histogram, layer bisection, and a sibling oracle
`loop`

When improving a scripted policy, most of the win is in HOW the agent moves and WHAT it targets, not goal selection -- a good goal picker that oscillates, freezes, or wanders never realizes its plan, so diagnose movement/targeting from the WORST episodes first and map each symptom (frozen, A->B->A->B looping, piling on a hub, wandering, dying, head-on collisions) to a module and knob rather than guessing parameters. Use the per-episode action histogram as a cheap health metric (no replay): healthy alpha_cog ~30% noop with balanced movement; broken run 75-80% noop; a symmetric/uniform distribution across opposing directions plus self-position cycling through a tight ~12px loop is the smoking gun for a path-follow ORBIT (distinct from wall-stuck); but allowlist benign single-frame mask=0 blips and the pre-game noop run by segmenting per phase. When intent looks right but movement is wrong, bisect layers explicitly: confirm the A* corridor, then intent direction, then the intent->action-index->button-mask mapping is byte-for-byte identical between policy and server protocol, only then suspect path-following. A working SIBLING mode sharing the same encoding pipeline is a diagnostic oracle: if it moves fluidly, the shared pipeline is ruled out and the bug is in the broken mode's logic (e.g. imposter mode had a 240-tick followee swap floor that crewmate mode lacked -- the missing commitment was the bug). #### 30. Track unseen Crewrift agents with a reachability disc and deterministic path-mixture belief
`loop`

Build a multi-agent position filter that is cheap, deterministic, and RNG-free by reframing the only real latents as (i) which tasks each agent is assigned and (ii) which it's heading to; given a destination the A* path and position are deterministic from the static map, so represent belief as a soft task-assignment distribution plus a destination mixture rolled forward along PRECOMPUTED A* polylines (no live search, no Monte-Carlo -- spread comes from the mixture, not sampling). Bound an unseen agent with a hard reachability disc: T ticks after last sighting it is within speed*T pixels (Crewrift ~2.75 px/tick), intersected with reachable cells -- exact, cheap, assumption-free, so use it as the backbone before any behavioral prior. Size the occupancy grid (~32px bins) to the EXPECTED path-prediction divergence, not arbitrarily fine, since polylines only approximate real paths and pixel-exact prediction would be falsely confident; the grid doubles as the error model and gets corridors for free. #### 31. Naive flee-target projection ignores layout and idles after arrival
`crewrift` · **negative result**

Computing a flee target by projecting the bot through itself away from the threat (fleeTarget = self + 2*(self - body)), clamping to bounds and snapping to the nearest walkable pixel, has a KNOWN WEAKNESS: it ignores map layout, so in a dead-end room the imposter flees toward a wall and wastes time on snapToPassable retargeting before a waypoint route is found -- fleeing toward the nearest room EXIT would be better (snap+waypoint handles most cases, treated low priority). #### 32. Check what the environment hands you for free before building search; and beware shared-action determinism
`loop`

Before building search/exploration for a game policy, check what GLOBAL information the environment already gives for free -- if a minimap already reveals the whole map and other agents' locations, an explore task is pure cost and should be dropped entirely. Most game states (lobby, role reveal, gameplay, deaths) are reachable WITHOUT navigation; only a few (e.g. voting at a button/body) need it, and navigation-to-trigger-a-view is a chicken-and-egg trap worth avoiding when a state is otherwise reachable. `loop`

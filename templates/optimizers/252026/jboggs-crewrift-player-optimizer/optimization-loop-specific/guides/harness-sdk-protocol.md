# Harness, SDK & wire protocol — guide (loop tier)

Reference notes, design rationale, and negative results (20).

#### 1. Derive RULEBOOK/GAME_API from engine source; protocol capabilities are asymmetric and porting math silently diverges
`loop` · ⚠ _session-derived, unverified_

Before writing any agent for a coworld game, derive two reference docs straight from engine source: a RULEBOOK (rules, roles, phases, win conditions) and a GAME_API (run the server, connect over the transport, interpret input frames, send actions). Verify protocol assumptions against the server's routing code, not docs, because send-path and read-path capabilities can be asymmetric (e.g. an agent confined to a sub-channel may be unable to READ a global channel yet still be able to SEND to it) -- check both directions. When a local harness pins a player's role/slot, trace it to the server (CLI arg -> a slots config field -> JSON -> server pins slot N), noting that a role-only pin may not mark the slot restricted, and that role parsers often accept multiple spellings case-insensitively. When porting projection/perception math between languages, implementations silently diverge (e.g. a Python perception did proper ray-clipping while a port merely clamped to bounds); diff a ported geometry helper against the reference and confirm which matches the server's actual placement algorithm before relying on it.

#### 2. Keep the perceive->belief->mode boundary clean: modes emit scratch signals, never mutate belief; reject the delta pattern
`loop` · **negative result** · ⚠ _session-derived, unverified_

Keep deterministic mode logic free of direct belief mutation: have modes emit only scratch signals (e.g. a failed-strike observation) and merge them into belief in a single envelope/belief-merge step, routing any new self-correcting signal through that existing path, so behavioral inference stays traceable and modes stay pure. Keep the strategy-facing belief object clean -- never stash raw transport state or pending trace events on it -- and keep core decision functions pure and sink-free by routing emission through a separate emitter/hook rather than threading a sink into every signature. NEGATIVE: do NOT adopt a delta/patch mutation pattern (every hook/mode/task returning a BeliefStateDelta) unless you have real concurrency or time-travel needs; its only real benefit is merging independent concurrent writes, which does not apply when execution is sequential. Prefer direct mutation of mutable state and get traceability from a thin logging proxy, because defining the delta language (deep vs shallow vs path merges, append-vs-replace, conflict resolution) is hard and exists only to avoid a direct assignment. Keep explicit atomic 'message' boundaries only where data genuinely crosses a concurrency/async seam (an async decide loop publishing a deep-copied (ModeDirective, belief_updates) tuple to the fast act loop); do not generalize that one serialization point system-wide.

#### 3. Belief state: flexible dict, namespaced inferences, world knowledge vs task scratch, reset clears ad-hoc attrs
`loop` · ⚠ _session-derived, unverified_

Make the policy's belief state a flexible nested dict with a small set of required fields (current_task, position, role, perception outputs) and everything else free-form, so each mode stashes custom data and new behaviors add without schema migrations. Give LLM-injected/outer-loop inferences their own namespaced sub-dict (an `inferences` field modes read) and keep framework-managed fields (tick, position, view, occupancy_grid) read-only to the outer loop so an untyped dict's top-level assignment cannot clobber machine-perceived state. Store persistent world knowledge in belief state and only per-task control state in task-scoped memory: 'areas I've visited' is world knowledge (reuse viewport-confirmed occupancy-grid cells) and must live in belief, because task-scoping loses it on any interruption and causes re-exploration. A reset() must clear ad-hoc attributes stashed dynamically at runtime (delattr the unknown attributes), not only declared fields, or transient state leaks across episodes -- and a snapshot-based reset must include the bookkeeping that drives it (ActionMemory.clear() omitted the _standard_fields registry attribute itself, wiping the very list that tracks what to clear). Maintain a documented registry of every extension/scratch key an agent stores on shared state (belief_state.ext), recording each key's type/lifecycle/purpose to prevent namespace collisions, distinguishing keys that persist across ticks from keys in a wholesale-replaced dict (which every returning code path must re-include -- a fragile pattern), and prefer persistent storage for invariants like hysteresis bookkeeping. In a coworld policy beliefs in wm.beliefs persist across ticks and are the right place for a per-miner lock like locked_mining_tags; clear it when cargo returns to zero.

#### 4. LLM calls must NEVER block the real-time frame loop; async dispatch, per-kind timeouts, drop stale/late results
`loop` · **negative result**

Make 'the per-frame control loop never blocks on an LLM call' the top hard constraint for any LLM-driven game policy. A 5-9s blocking call inside the per-frame loop gets the bot dropped even when the call succeeds: while it blocks the server keeps streaming frames, the client's socket receive buffer overflows (~170 frames > 1MB vs typical 64-256KB OS buffers), the server's send fails and drops the player, and the next receiveMessage hangs forever (symptom: bot freezes repeating its last action). 'Another bot blocks and votes fine' is no proof: compare per-call duration to socket send-buffer headroom, not sync-vs-async (a ~1-3s in-process libcurl call survives; a ~5-9s fork+exec subprocess does not); blocking only survives confined to an idle phase (the voting interstitial). Budget ~50 LLM calls/game at ~2-4s each and dispatch asynchronously, consume the result on a later frame, allow only one pending output at a time, and if it misses the per-turn budget (~4s listen window) fall back immediately and DISCARD the late result. For many agents dispatch each as a non-blocking future via a ThreadPoolExecutor sized to the agent count and gather at a wall-clock deadline (MODTALKS_LLM_DEADLINE_SECONDS default 12s), leaving slow futures IN FLIGHT (providers rarely honor cancellation) -- measured: 8 concurrent Bedrock agents cut p50 ~33s to ~9.2s. Set per-call-kind timeouts (hypothesize/strategize 20s, react 15s, accuse/persuade 10s); retry ONLY transient failures (429, APITimeout, APIConnection, 5xx) with bounded backoff (max 2 retries 0.5s/1.5s), never deterministic 4xx, and abandon a retry whose backoff exceeds the deadline. Discard stale responses by comparing dispatch-time stage against current stage. Enforce context size with progressive multi-tier trimming dropping lowest-value sections first (halve sightings, then chat, then full chat log, then prior meetings, evidence scores LAST), newest-first. NEGATIVE: provider latency can structurally EXCEED the per-turn budget (Bedrock p50 ~9s vs a ~4.2s vote window) so the async result is almost always too late -- async dispatch alone does NOT solve latency; widen the budget or rely on the fallback. Measure provider p95 against the phase tick budget on the REAL production provider (cold live Bedrock), not a warm dev harness. Capture LLM context byte count at DISPATCH time (capturing at response time reads zero because dispatch may have cleared the buffer).

#### 5. Emit reporter event-logs as JSON not parquet; events.parquet is four columns with JSON values reconciled to results.json by slot
`loop`

The {ts,player,key,value} parquet event-log layout was legacy/superseded (banner-marked in artifacts/EVENT_LOG.md) where the current output_format is {mime, schema} with schema being JSON Schema, which doesn't fit binary parquet -- emit event rows as JSON (it fits the contract and is nearly free in Nim, whereas parquet needs a hand-rolled writer; the old scribe reporter hand-rolled ~308 lines of parquet.nim). Where the Coworld events.parquet schema IS the contract, it uses EXACTLY four columns -- ts (int64 episode tick), player (int64 slot, or -1 for global; the canonical 'player' column dtype is int64 NOT pa.int16(), and pre-canonical int16 round-trips locally so check dtypes explicitly), key (string dotted event name), value (string, JSON-encoded even for scalars). The contract FORBIDS extra columns (subject_player, x, y, phase, round); put all such structure inside value JSON. Use dotted namespaced keys (lifecycle. roster. action. vote. discussion. outcome. summary. state. meta.) so consumers filter with key LIKE 'vote.%'; keep prefixes game-agnostic and put game nouns in the suffix/value. Version with one global meta.schema_version row (ts=0, player=-1); evolve ADDITIVELY. An empty log must still be a well-formed zero-row Parquet table, Snappy-compressed in a single row group, rows pre-sorted by (ts, seq, key) for byte-determinism. Per-slot summary.* rows must reconcile EXACTLY with results.json. Tie every Crewrift event to the policy by 0-based SLOT INDEX, never raw color/name: stdout COLOR/name, results.json PlayerN arrays, and config slots[] all reconcile to slot, and the runner assigns slot i the token config slots[i].token. Split by access pattern: events.parquet is authoritative for flat cross-episode analytics, a companion trace.jsonl holds nested/high-volume data.

#### 7. Design match-request schemas as a per-seat discriminated-union model; trace every field end-to-end before claiming it is load-bearing
`loop` · **negative result** · ⚠ _session-derived, unverified_

A roster/matchmaking request grown by accretion (explicit policy_version_ids, then named-player and top-N selectors) ends up with fields meaningful for some modes and ignored for others, and offers two mutually exclusive ways to say 'who am I': a direct caller-owned policy-version list (allows explicit slot assignments but forces the whole roster to be yours) vs a league 'requester' selector (pulls in opponents but forbids explicit assignments), so you cannot mix your bot with others' and control seating in one request. The clean fix is a unified per-seat model: a list of participants each carrying a discriminated-union select (policy_version | player | top_n | random) plus its own count and optional slot. Pin distribution semantics explicitly: round-robin is ACROSS the N episodes not within one; a fixed slot occupies that slot in all N games (only sensible with count==1). NEGATIVE: before claiming a field is load-bearing, trace it end-to-end -- an assignments int[] column was NOT NULL with empty default, written empty, read NOWHERE on the live dispatch path, and a per-episode seed was computed/stored but never passed to the runner; when documenting combination rules, read the BACKEND VALIDATOR not model docstrings (they drift and over-state exclusivity: at runtime only 'requester' is mutually exclusive with an explicit roster while 'opponents' and 'top_n' CAN be appended).

#### 8. Coworld CLI footguns: prefixed identifiers, inconsistent list shapes, three index spaces, and silent-stall launch traps
`loop` · **negative result**

Coworld v2 backend identifiers (LeagueId, DivisionId, RoundId, EpisodeRequestId) are Pydantic PrefixedId subclasses validated only against ^<prefix>_<uuid>$, so FastAPI returns HTTP 422 BEFORE the handler for any slug/short-id; only --policy NAME:vN works name-based -- resolve leagues/divisions client-side by listing and matching when the arg isn't prefixed. List endpoints return INCONSISTENT shapes: only `rounds` returns a paginated envelope {entries,total_count,limit,offset}; every other list returns a bare array and silently truncates at `limit` with no total_count, so branch per command. A single episode juggles three integer index spaces -- slot index (the N in --agent N and policy_agent_N.txt, 0..num_agents-1), compacted policy position (index into the deduped policy_version_ids list, what participant.position/assignments[slot] refer to), and pool-entry seed_order; compose `assignments` with `participants.position` to map slot to policy. `episode-logs --agent N` only validates ownership when `--mine` is passed, so without it you can silently read another player's log. #### 9. results.json scores is the only cross-game guarantee and the scoring source of truth
`loop`

In the Coworld results contract, only the `scores` array (one number per slot) is guaranteed across ALL games -- game-specific fields (crew/imposter/tasks/votes) live in each game's own results/manifest schema and must NEVER be assumed by game-agnostic platform code. results.json is the scoring source of truth: any event log or summary must reconcile to it EXACTLY and never contradict it. #### 10. Design tracing as orthogonal concerns; the player uploads its own size-capped artifact; hosted logs are tail-truncated stderr
`loop`

Design player tracing as three orthogonal concerns (event selection, output routing, encoding format), not a hardcoded 'stderr JSONL' sink: keep the domain event/metric API tiny (events, counters, gauges, histograms) and add a separate output layer composing format@destination specs (jsonl@stderr,csv@artifact,parquet@artifact). The hosting platform does NOT scrape the container filesystem; the player must write and upload its own artifact to the runner-provided destination via env (a presigned S3 PUT hosted, a file:// path locally), creating one size-capped zip and uploading before teardown with the trace finalizer closing/uploading after the main loop ends. Hosted policy containers capture ONLY stdout and stderr, merged into a tail-truncated per-slot file with no file upload and no pagination, so any JSONL-to-local-disk trace produces nothing in tournament logs -- emit tracing to stderr from the start and budget volume: at ~24fps one decision line per frame over a 3-minute match is ~4300 lines (~860KB) while full per-tick tracing (~16 lines/tick) blows past the captured tail, making early-game events unrecoverable; gate verbose traces behind a flag, pick the LOWEST level that answers the question, or capture full logs locally. A trace system that advances its endTick/manifest only on WRITE freezes its reported end time at the last logged event. A clean WS close plus a startup line is NOT evidence the policy played well -- the usual failure is rich infra logging while the policy's own reasoning emits nothing; instrument WHY a mode/branch was chosen, beliefs (where partners/opponents are, urgency, goal flags), and FSM transitions, using a hybrid scheme (emit a strategic-state-change event only when a watched field differs from the prior tick, EXCLUDING monotonic counters, plus a full snapshot once per game-second). Separate the bot's internal queryable memory (consumed by the policy in real time) from the append-only trace/event log (read offline): different lifetimes and consumers.

#### 11. Use the engine's ground-truth tick as the correlation key threaded through the SDK in ONE place; never a local counter
`loop`

Use the ground-truth tick the game streams (Crewrift now streams a constant tick counter, a 'tick <N>' marker sprite) as the correlation key in perception, belief, and ALL tracing -- never the player's self-incremented counter, which drifts -- and put it in every log entry so reasoning events tie back to the recorded frame. The players.player_sdk AgentRuntime keeps its OWN tick counter and ignores observation.tick (step() does self.tick += 1; self.emit.tick = self.tick; perceive(observation, self.tick)), so to thread the engine tick through perception, belief, mode/directive timing, and every trace/metric in ONE place set runtime.tick = server_tick - 1 before step() (it increments to the server tick); changing only perceive() would fix belief but NOT tracing. A slow outer loop holds a snapshot that LAGS the live tick, so measure staleness by injecting a live-tick accessor (tick_provider) into the outer loop constructor (staleness = tick_provider() - belief_state.tick). Don't trust the simulator's last_action_success flag alone for stuck detection -- it can disagree with the observation's last_action_move token (an agent reported move_south success=True yet stayed put for 5875 ticks); trust the global observation token (last_action_move, surfaced as last_move_succeeded), and with a two-phase move handshake the attempt is recorded on tick N (no position change) and verified at the start of N+1 before updating spatial.position, so any state derived from position must run AFTER that verification. Parallel agents don't share a tick clock: treat event ticks as per-agent with a small offset when cross-referencing logs.

#### 12. Build memory as three subsystems (spatial, declarative, temporal) with read-only spatial/temporal; flexible dict belief
`loop`

Structure an agent's persistent state as three subsystems: spatial memory (a global map accumulated from egocentric observations), declarative memory (a writable fact store of beliefs/goals/agent-state), and temporal memory (ring buffers/EMAs/tick-stamped snapshots tracking change over time). Make spatial and temporal read-only from the decision layer (only perception updates them); let the decision layer write only to declarative memory. When a per-tick decision needs multi-tick state (persistent-deficit detection, stuck detection, success-rate-per-target, memory staleness), build the reusable temporal layer ONCE rather than bolting a bespoke counter onto each hypothesis -- several mining experiments independently hit the same single-tick-signal limitation that only this infrastructure resolves. Run a post-belief-update hook each tick to detect behavioral patterns invisible to instantaneous perception (aggressive probing, going passive after urgency): feed raw observations into per-player accumulators, derive flags on threshold crossings, then run soft inference; snapshot cross-round summaries before resetting per-round counters and preserve interaction-history fields across rounds. For a per-tick memory-debug TUI, get the wiring order right (memory.update(parsed_obs, last_action, obs.last_action_move) first, then render, then wait for input, then store the chosen action) and construct the Memory object via the shared compound-bootstrap path so the tool is a faithful view of agent state.

#### 13. Match agent phase-transition timing to the server's actual state; gate phases off belief flags not raw ticks; scope caches narrowly
`loop` · **negative result** · ⚠ _session-derived, unverified_

Agent-side phase timing must match the server's actual state -- agents opened solo whispers right after role reveal because the intro-panel confirm landed after the server had already entered Playing, so acting before the playing phase truly begins wasted actions. #### 17. Keep the policy pure and I/O-free; standardize a per-agent subprocess entry point discovered by folder convention

#### 20. Expect scripted policies to plateau ~top 10% against learning competitors; the gap is mining, disruption, and coordination

# Navigation & pathfinding — recipes (loop tier)

On-demand recipes (37). Trigger→action heuristics; pull the relevant one when its situation arises.

#### 1. Block ALL interactive-object cells in A* and path to a neighbor + bump
`loop`

Routing A* transit THROUGH interactive-object cells (extractors, hubs, junctions, gear stations) causes catastrophic stuck loops: moving into an oxygen_extractor triggers its on-use handler and consumes the move WITHOUT displacing the agent, so 'successful' moves never change position (observed stuck 5875 ticks). Excluding only the target's own cell is insufficient because the path can route through OTHER intervening objects. Fix: add ALL solid interactive-object cells (plus currently-visible non-self agents) to the A* blocked set at path-COMPUTATION time only -- in alpha_cog _navigate_astar the blocked set = confirmed walls + interactive_cells + visible agent positions -- but do NOT remove them from the passable-cells topology, because those cells are walkable after interaction and an agent standing on one must be able to leave. EXEMPT the goal cell from the blocked set so A* can still path toward an occupied target. Then path to a cardinal NEIGHBOR via a multi-goal A* (astar_path_to_adjacent terminating when any cardinal neighbor is reached, target excluded) and emit ONE directional bump to interact (the move fails as a move but triggers the object's handler); the empty/already-adjacent path MUST fall through to that bump or the agent does nothing. This dropped max stuck time from 5875 to 11 ticks. When a shared helper blocks ALL tagged cells (including agents), strip agent tags explicitly before pathing through transient obstacles. After the bump, clear ALL cached path state so the next tick re-evaluates fresh. To add a new avoidance (danger cells), extend the blocked-set construction, not the A* heuristic.

#### 2. Two-tier waypoint graph: strategic A* over waypoints, tactical A* per tick
`loop`

Advance the path-following index only when the agent's position matches the expected next cell (confirmed via the simulator's last-move-succeeded signal); on a failed move, keep position and index unchanged and retry, preventing path corruption and letting transient agent-blocking resolve. Make A* agent-aware: add currently-visible peer positions to the blocked set as TEMPORARY obstacles for that one computation (never permanent walls, never guessed unseen positions) -- without this A* routes through teammate-occupied cells and agents burn 26-54% of ticks failing to move at chokepoints. On a failed move distinguish a wall collision (record the cell, replan immediately) from a transient agent blocker (only force a replan after 2 consecutive blocks). Critical: retry-then-sidestep-then-replan CANNOT break a chokepoint loop if the replan is blocker-UNAWARE -- it recomputes the same path through the blocked cell and oscillates. Working recipe: retry up to 2 ticks, deterministic right-relative sidestep (N->E->S->W) so two head-on agents split with no coordination, then replan FROM THE NEW POSITION with the blocker added.

#### 3. Detect oscillation as its own failure class with NO distance gate
`loop` · **negative result**

Oscillation is a distinct failure from slow progress and from frozen-against-blocker, and a detector based purely on 'position unchanged for N ticks' is too weak -- agents oscillate while CHANGING position every tick (a trapped aligner bounced between two cells; a miner's moves all succeeded yet it never reached its goal). Run THREE detectors in parallel over a position ring buffer: (1) position-stuck (position unchanged ~8 ticks) catches frozen-against-blocker; (2) pacing/cycle-stuck -- position-history cycle matching against period-2/3/4 cycles over a ~12-tick buffer (or confined to <=2 unique cells over the window), firing after ~5 cycles / ~10 ticks with NO distance gate -- catches A->B->A->B oscillation, which moves every tick (X changes, Manhattan distance dips each pass) and resets any per-tick stall counter; (3) axial/distance-staleness -- distance-to-target over a rolling ~80-120 tick buffer stops decreasing -- catches moving-but-not-progressing toward a far goal and distinguishes a true trap from a wall-impeded agent making legitimate slow progress. A single detector misses the others. Detect lack of progress TOWARD THE GOAL over a window, not per-tick position change. Axial-stuck is target-shaped, so keep it OUT of the auto-tabu/auto-blacklist wrapper (callers usually want to re-route to the same goal); only position- and pacing-stuck should auto-blacklist the target. When a gated mechanism 'never fires,' verify the GATE is not the problem before concluding the mechanism is unneeded; gate unsticking on 'about to arrive' (very small distance AND a viable path exists), not a single distance threshold.

#### 4. Stuck recovery must DISPLACE the agent; gate by distance band + second-stuck + TTL blacklist
`loop`

For oscillation/stuck recovery the recovery ACTION is the bottleneck, not detection -- a correct distance-to-target detector still regresses if the action disrupts productive agents. Soft recovery (rotating heading 180 degrees, or a single-tick nudge) is DEAD against greedy-nav oscillation: every soft-rotation arm produced worst-in-class A/B means (~-1.50) because greedy nav just picks a different oscillation axis. What works is a bounded ~5-tick blind explore burst that physically DISPLACES the agent ~3-5 cells, changing its observation window enough to re-pick a different target; design recovery around how much spatial displacement changes the agent's information state. Gating: fire the burst only when the agent is >10 cells from its target (the distance gate suppresses false positives near the target, wall-impeded but legitimately close). The >10-cell gate + 5-tick burst turned an unshippable detector into V7: 34W/8T/8L +1.86 mean vs V6 on 50 seeds at 10k, single-seed deltas up to +7.95/+4.73. Use second-stuck gating: on the FIRST stuck use the cheap blind-explore; only on a SECOND stuck on the SAME target trigger aggressive blacklist+re-pick, so 'star seeds' (where staying stuck on the best target beats a worse one) don't collapse. The blacklist must PERSIST with a TTL (~60 ticks = window/2 worked; window/4 ~30 ticks oscillated and was much worse; 1-tick is useless). On re-pick, accept an alternative only if no farther than the stuck target (+0.89 -> +1.38). Shipped as V8/H048: 30W/11T/9L +2.23 mean, the largest single-flag improvement in the project. Refinement: an UNCONDITIONAL displacement burst on every second-stuck repaired three loss-tail seeds but catastrophically regressed two top-win seeds; naive distance-THRESHOLD gating was too coarse (D20/D25/D30 byte-identical), and only a tight distance BAND (30<=stuck_dist<=35) preserved the repairs while skipping both known large-regression triggers (V9/H058, 4W/44T/2L +0.345). Note miners keep the unconditional V7 explore because deficit concentration is load-bearing for that role.

#### 5. Use sticky targets with progress invalidation to stop per-tick re-rank flips
`loop` · **negative result**

Goal oscillation between two near-equidistant targets comes from a target-selector that re-runs best-pick (min) fresh every tick with no memory; each flip resets the goal, wipes the A* cache, forces a replan, and yields net-zero progress (one trace: 23 goal runs / 22 flips over ~638 ticks), and it is especially wasteful under momentum (Crewrift) where the agent slides back and forth. Diagnose by logging the chosen goal per tick and counting distinct goal 'runs' -- rapid flipping is the signature, distinct from wall-stuck or coordinate bugs. Fix with a STICKY target: commit to the chosen target for a minimum window (e.g. swgy_targets/progress_window=12, or TASK_COMMIT_TICKS=48 ~2s at 24 FPS) and snap the destination exactly once (guard with a flag like fleeCoverSet). CRITICAL: a sticky target WITHOUT progress invalidation traps the agent forever on an unreachable target -- always pair it with a progress check that drops it after N ticks of no strictly-closer distance; the progress half is the half that matters. When you do allow re-evaluation, gate the switch on a shouldSwitch predicate that resists oscillation -- switch only on a legitimate reason (target resolved/vanished) or a strict objective UPGRADE (higher value/tier via a fixed priority ladder, e.g. visible on-screen icons beat radar pointers to offscreen tasks) OR a same-tier candidate significantly closer (distance < current * 0.5, a named constant); a bare 'closest wins' thrashes. Throttle re-evaluation to every N ticks (store last re-eval tick), keep commit-window length and re-eval period as two separate documented constants, and scope re-selection to the navigation/approach phase only, never while actively executing the objective (Hold/Confirm). Reuse the existing mode object when the incoming directive still matches_directive so mode-local state (phase, target commitment, waypoint index) is not cleared.

#### 6. Treat an empty path as UNREACHABLE; never silently emit zero or wall-steer
`loop` · **negative result**

A pathfinder that rejects an impassable goal and returns an empty path is a latent DEADLOCK -- if findPath early-returns [] when the goal fails passability and movement is guarded by 'if path.len > 0', the bot emits zero movement forever. Equally bad is collapsing an empty route into 'drive straight at the goal' or a greedy direct-steering fallback: it masks the failed plan while bouncing off walls. Always pair 'reject impassable goal' with EITHER a fix to the plan/graph OR a safe fallback -- and the good fallback when A* returns no path (all known routes walled off) is a DIRECTED explore toward the goal: move in the target's general direction preferring unseen cells, filling spatial memory until a path becomes discoverable or the window sees the target, and let anti-stuck jiggle handle walls (a sibling policy, modulabot/policies/base.py, survived a map a non-fallback policy was paralyzed on precisely because of this -- a degraded-but-moving behavior beats a clean-but-frozen one). Never emit silent zero-movement and never an unsignalled wall-drive: emit an explicit stall/replan signal and give stuck/stalled tasks a status/progress channel back to the policy (not just (memory_update, action)) so it can trigger re-pathing or a strategy change; without it, escalation is impossible.

#### 7. Fix waypoint-follower wiggle in selection/snap, not the steering symptom
`loop` · **negative result** · ⚠ _session-derived, unverified_

When a waypoint-follower wiggles in one spot, the root cause is usually nearest-waypoint selection using Manhattan/Euclidean distance only, IGNORING walkability, so it picks a waypoint just across a wall; fix the assignment (reachable snap point, or walkability/line-of-sight-aware nearest-waypoint), not the steering. Snap a follower to the nearest path SEGMENT (Euclidean projection) and compute progress as interpolated distance ALONG the segment, not the nearest VERTEX: after DP simplification consecutive vertices can be far apart (a 91px gap with a 30px snap radius), so a bot midway along a long segment is beyond every vertex's snap radius, the 'out of snap range' branch fires, a resync steers it to the segment start, and it oscillates forever; fix this in the perception/lookahead layer (segment-aware snap plus interpolated lookahead in selectLookahead). Path-follow oscillation (distinct from wall-stuck) also comes from a large lookahead combined with localization jitter and aggressive waypoint trimming on a path computed once and never recomputed: with PathLookahead=18 near a corridor turn, trim logic (dropping steps within distance<=2) consumed nearby waypoints so the lookahead pointed PAST the turn across a wall and steering reversed. Fix: (1) shrink lookahead (18->4 or ~8) so steering hugs the A* corridor through turns; (2) periodically recompute the path (every ~24 ticks) so jitter-corrupted trim state self-corrects; (3) add a progress-stall replan if distance-to-goal hasn't decreased in ~48 ticks. Separately, staccato (move-pause-move) means the bot runs out of steering target between expensive replans -- DECOUPLE recompute cadence from steering-point selection: throttle A* to the replan interval but every tick advance a moving lookahead point along the already-stored path. Watch for a 'dead zone' (a walkable region with NO waypoint) which strands any agent that replans there; add a waypoint with edges and re-bake. Audit that tuning constants are read at their use site (a JIGGLE_TICKS knob was silently overridden by a hard-coded literal).

#### 8. Multi-agent navigation: stagger doorway approaches and break symmetry on rendezvous
`loop` · **negative result**

When several agents must approach the same one-cell station doorway (opening rush or shared equip), pass an approach_index / stable approach slot so they target different station-adjacent cells, and stage overflow agents on diagonal cells until a route opens -- this prevents duplicates from blocking each other before bumping the station. Two cooperating agents running the same greedy policy can defeat a shared goal (both key partners independently 'created a new whisper' when a partner was nearby, splitting into two solo rooms that never merge): break symmetry with deterministic opener-vs-requester roles for known pairs. In FOG-OF-WAR games rendezvous is the binding bottleneck: across Persephone/Eurydice self-play, agents learned roles and started many probes (72 targets, 41 probes) yet full 10-agent live runs DREW because only ~6 probes completed and 34 hit initiate_timeout with ZERO server-confirmed joins -- the minimap doesn't show whisper status and speech bubbles are visible only within ~30-40px, so by the time an approacher sees a creator's bubble the creator has often timed out (~15s), yielding separate solo whispers; a fixed-coordinate exchange protocol (requesters pulse near a key whisper while openers run blind GRANT windows) ALSO failed across eight server logs. What works is the 'stupid meetup': each ally shouts a FIXED coordinate (e.g. 'meet at 50 50'), all walk there, first to arrive opens a whisper others auto-join -- converging on a known point beats mutual target-chasing (each chasing the other's last-seen position never converged; the minimap self-dot overwrites others). Design coordination around visibility/timing (linger longer, approach to within bubble-visible range BEFORE deciding to create vs join, announce intent on a global channel), not more perception fixes.

#### 9. An energy/cost-stalled failed move is indistinguishable from a wall block; detect intended!=executed and retry
`loop` · _see related: U0611 (other tier)_

In a coworld sim where movement has a resource cost (energy) and the engine executes a noop when the cost can't be paid, a stalled failed move is INDISTINGUISHABLE from a wall/agent block in the executed-action token — a navigator that ignores the resource will misread exhaustion as obstruction and thrash. Rather than building heavy resource-management logic, detect the failure (intended move != executed action) and RETRY up to a small cap (e.g. 3), since agents typically auto-regenerate the resource each tick. Trigger retreat on distance-aware vitals (e.g. retreat if a survivability resource < distance-to-safety + margin) and only resource regen that is NOT effectively instant near friendly structure needs a rest ceiling; resource-aware behavior (resting to refill instead of burning failed-move ticks, biasing targets toward regen zones) is a large untapped navigation lever.

#### 10. Exploration: deterministic coverage routes, share map before adding scouts
`loop` · **negative result**

Do NOT separate exploration from the productive objective into distinct modes, and do NOT use reactive frontier exploration that re-decides targets each tick: a frontier-picker that only targets cells adjacent to unexplored space keeps agents trapped in a small bubble near spawn (e.g. ~6% of an 88x88 map after 2000 ticks). UNIFY explore with the objective and assign deterministic space-filling/patrol routes — move outward in a boustrophedon/quadrant-snake and capture/interact en route. For pure coverage, snake-scan in strips of width = one observation diameter (e.g. 12 for a radius-6, 13x13 window): assign each agent a quadrant and walk it boustrophedon, giving full coverage in O(area/observation_width) with overlapping vision. For coordinated exploration WITHOUT shared memory, assign exits/directions at the pre-match staging point where agents are co-located (each picks the nearest base exit, tie-broken by agent ID, then spirals outward) instead of relying on not-yet-filled spatial memory; bias the initial direction toward likely structure. NEGATIVE RESULT: do NOT add a scout role (or have producers 'explore') when there is no shared team map — discoveries can't propagate; either make map knowledge shareable first (e.g. a talk channel when agents meet) or fold that body into another role.

#### 11. Contention: precompute worst-case arrival; pickup full loads from known capacity
`loop`

When other agents contend for the same resource, replace fragile per-tick timing triggers with a worst-case precomputation: compute the maximum possible travel time, hardcode that buffer, and arrive early once — avoiding race conditions and simpler to reason about. For resource-pickup loops, compute how many items to take up front from a known capacity/maximum rather than the 'keep taking until the count stops incrementing' poll-until-no-change hack that causes wasteful round trips; let one carrier grab a full load in ONE greedy trip (strict fairness isn't worth the cost — one agent filling to capacity beats several making single-item trips). Use threshold comparisons (>=), never equality (== capacity), for quantized resource checks — e.g. a miner with capacity 44 mining in increments of 10 never hits exactly 44, so an '== full' deposit trigger silently never fires. A carrier that reaches a shared deposit with nothing to carry should wait one tile AWAY, not idle adjacent, because adjacent idling blocks other agents trying to deposit.

#### 12. Drive menu/cyclic navigation from perceived cursor with press cooldowns
`loop`

Menu navigation over a laggy client needs press cooldowns, not per-tick re-presses: a MenuNavigator pressed B to open a menu, but it took 2+ frames to appear so on the next cycle it still saw the default bar and pressed B again, toggling the menu closed -- after an open/navigate press, wait a fixed number of ticks for the UI to reflect it (or use a verified fixed sequence like B -> Right x3 -> Down x2 -> A when the layout is stable). Separate 'where the cursor is' from 'what button to press' and drive from the cursor READ from perception; only fall back to an internally-tracked cursor when perception cannot see it, and then advance the remembered index only on rising-edge ticks where a press actually fires (one per tick), or it desyncs. Navigate a cyclic selector (menu ring, L/R picker, U/D/L/R grid, or the voting cursor as a WRAPPED ring of playerCount+1 slots -- player slots plus a SKIP slot) by computing the SHORTER of the two cyclic distances and stepping that way (index 0 adjacent to N-1; for 8 voting players the worst case is 4 cursor moves); always-advance-one-way wastes step-limited actions. This needs the full ring order as a known constant -- ground it in the game's own spec (GAME_API.md), not an invented parallel ordering, and infer UI grid geometry from live counts (row = i // cols, col = i % cols, rows = ceil(count / cols)). Separately, maintain a self-position keyframe history (capped, skipping low-information corridor positions, exposed newest-first) so an agent can later truthfully report where it was and a deceiver is constrained to claim only locations it actually visited.

#### 13. Run A* over the full remembered map (not the obs window); it is cheap, so replan freely
`loop`

Navigation experiments confined to the agent's 13x13 (169-cell) observation window -- full BFS, biased BFS, territory-aware nav -- all regressed or were byte-identical, because within that tiny window greedy and BFS pick the same first step. Real gains require A* planning over ALL remembered passable cells (thousands by mid-game). Use a Manhattan heuristic (admissible/consistent for 4-connected grids) and treat unknown cells (not in the known-walls set) as PASSABLE so the planner routes through unexplored space; only known walls are permanent obstacles, and stuck-detection re-pathing handles undiscovered walls. This is cheap, not expensive: on an 88x88 grid A* costs roughly 5-20 calls per agent per 10000-tick episode, each well under 1ms with a Python heapq (Crewrift's denser graph is ~0.2ms/call), negligible against per-tick observation parsing. Replanning is therefore nearly free, so re-root the route at the agent's live position every ~8 ticks and whenever the goal changes; this is what eliminates residual approach-wedging at the destination.

#### 14. Snap goals to passable cells and own grid<->world conversion in one module
`loop` · **negative result** · ⚠ _session-derived, unverified_

Always snap a navigation goal to a passable cell before handing it to the pathfinder: a projected target (e.g. a 'flee away from X' vector) can land inside a wall, A* returns empty, and the agent falls back to jittery greedy mode -- a snapToPassable on every computed goal removes that whole jitter/noop-lock class. When a coarse cell covers both floor and wall pixels, resolve toward the class the consumer least tolerates: apply WALL markings after FLOOR markings so a mixed cell becomes WALL, because a missed wall routes the planner into an obstacle. Have the planner return waypoints at the CENTER of each cell in WORLD coordinates and keep grid<->world conversion owned by the grid module (not duplicated in planner/steering); account for grid downsampling (multiply grid coords by the resolution factor, here ~2x, before A* validation) or off-by-resolution bugs make waypoints fail reachability for no apparent reason. Reject degenerate world dimensions (non-positive room_size from a bad/early frame) before building the grid, or you get a malformed grid and silent pathfinding failure.

#### 15. Replan A* only on meaningful events, and perturb inputs on a forced replan
`loop`

Re-plan a path only on meaningful events (target changed, wall discovered on the path, agent sidestepped off-path, path completed), NOT on transient agent-blocking or on observing new passable cells that don't affect the current path -- keeping the path stable across transient blocks avoids thrashing. CRITICAL corollary: a stall/replan timer that re-runs pathfinding with the SAME endpoints reproduces the same failure (an identical still-impassable goal returns an identical empty path), so any forced replan must perturb its inputs (exclude the previous endpoint, try a new target) or it is a no-op loop. For static-goal intents, cheap periodic re-planning every 4-10 ticks re-roots the path at the live position (the kill intent already replans A* every tick at 24Hz with sub-ms steps).

#### 16. Advance path index on confirmed moves; distinguish wall vs agent blockers
`loop`

Advance the path-following index only when the agent's position matches the expected next cell (confirmed via the simulator's last-move-succeeded signal); on a failed move position and index stay unchanged and the agent retries, preventing path corruption and letting transient agent-blocking resolve by retry. #### 17. Make path simplification wall-aware and bounded inside the recursion
`loop` · ⚠ _session-derived, unverified_

Any geometric path simplifier (Douglas-Peucker) on a baked grid path MUST be wall-aware: plain DP collapses a polyline whenever intermediates are within tolerance of the straight start->end line, but a wall-hugging path can be sub-pixel-straight while the simplifying line cuts through a wall. Before collapsing, Bresenham-walk every pixel of the straight line against the walkability mask and force a subdivision if any pixel is non-walkable. Enforce the max-segment constraint INSIDE the recursion as an ordered decision -- (1) deviation > tolerance: split at worst point; (2) length > max: split at midpoint; (3) wall crossing: split at midpoint; (4) else collapse -- NOT as a post-process, because post-process interpolation invents synthetic coordinates not on the verified A* path that could cross walls. A long gap between consecutive waypoints starves a reactive follower (it drifts/sticks on long straights), so keep a max-distance-between-points constraint (e.g. 30px) while keeping real A* coordinates. #### 18. Steer momentum-aware in sims with velocity/friction, not sign-function

#### 24. Avoid danger at plan-time via finite A* cost; measure productive output per life

#### 26. Grid-aware exploration: prefer UNKNOWN cells with a fallback chain to blind sampling

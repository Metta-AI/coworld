# Harness, SDK & wire protocol — recipes (loop tier)

On-demand recipes (47). Trigger→action heuristics; pull the relevant one when its situation arises.

#### 1. The wire protocol is game-owned: read game.protocols.player and never assume a shared SDK bridge
`loop`

Coworld does NOT define the player wire protocol -- each game does, so read the URI in the manifest's game.protocols.player BEFORE writing any player code, link it from the README, and resolve transport discrepancies before assuming the SDK JSON bridge applies. Examples differ wildly: Among Them/BitWorld speaks binary bitscreen_v1 (8192-byte 128x128 4-bit-packed frames in; 2-byte button packets or [0x01]+ascii chat out) and MUST NOT reuse the SDK's coworld_json_bridge; crewrift speaks binary Sprite-v1 screen-streaming; Cogs vs Clips uses JSON coworld.player.v1 (player_config then observation, reply with action by index/name); Paint Arena uses custom JSON. A scripted policy and a runtime-client policy are different architectures even when they share a name (a stale among_them screen-space scripted AgentPolicy for an in-process mettagrid engine was dead under the Docker/WebSocket spec -- grep the whole repo to confirm nothing imports a stale adapter before deleting it). Binary-wire games must own their own websocket bridge (e.g. coworld/policy_player.py invoked directly by the image CMD). Support multiple inbound formats at the entrypoint (detect raw binary on the first frame), pin protocol handling to a specific runner.py commit SHA, and re-verify at each phase boundary because drift is real. The config_schema must require 'tokens' as a string array with equal minItems/maxItems = slot count (variants omit tokens; the runner injects them).

#### 2. Bring up a new player wire-protocol-first: random-action loop, fps liveness, wall-clock pacing, verified endpoints, act on latest frame
`loop` · ⚠ _session-derived, unverified_

Before any strategy, prove the perceive-act loop with a 'random action' loop: connect, decode the latest frame, pick a random valid action, send it, repeat -- strategy is meaningless until wire protocol, frame decode, and action encoding work end-to-end. Detect joins by liveness, not handshake: the server may be silent on connect with no ack, so log the bot's own observed frame rate because absence of frames (not a connection error) is how a failed join shows up. Use wall-clock time for human-facing pacing (interstitial dwell, voting windows) and ticks only for sim-relative logic, because nominal 24fps and observed ~22fps drift. Verify connection endpoints in client source not prose docs (global view defaults to /global, player perspective to /player; how_to_make_a_bot.md was stale). Real-time perception/policy bots fall behind silently: if a step exceeds the per-frame budget the harness drops frames and the bot only sees the early, useless part of the episode -- at ~55ms/agent x 8 agents (~440ms/batched step vs a ~42ms tick) one bot processed only ~130-164 of ~1147 frames and NOOPed; compute throughput = frames_processed/wall_seconds vs tick rate. A receive loop that drains the whole socket FIFO (while ws.recv()) instead of taking only the latest frame makes the policy act on stale state when the client is slower than the server (~9.4fps client vs 24fps server); use the runner's latest-frame drainer (_receive_player_frame, 0.0 timeout, returns newest frame plus frame_advance count). Smoke-test cogames via `cogames play -m <mission> -p <policy> -r <log> -s <step_limit>` with a small -s for fast iteration.

#### 3. Treat the Coworld player container as a stdout-protocol, clean-exit, single-URL black box driven by env
`loop` · **negative result**

A Coworld player container sees ONLY a fully-formed engine WebSocket URL (host, slot, token, game params like role=scout embedded), its declared public manifest env, and per-version secrets -- never the manifest, game config, variant id, slot count, other players, or the Observatory backend. Read the runner-set COWORLD_PLAYER_WS_URL first then fall back to the generic COGAMES_ENGINE_WS_URL (full ws://...:8080/player?slot=<slot>&token=<token>), and use the URL EXACTLY as given (never strip/reorder query params, rewrite host, or hardcode slot/token) or the game rejects the connection; derive your slot from the URL ?slot=N, never from assignment/position arrays (four distinct numbering schemes with no documented mapping). COGAME_LOG_URI is optional and may not be injected by local Docker or hosted k8s (re-check runner source). Use stdout STRICTLY as the wire-protocol channel and route ALL logs/traces to stderr (audit deps for stray print()). Exit 0 cleanly on game-end -- there are no start/end messages, so detect a {"type":"final"}-style discriminator (the field is literally named 'final') or the socket closing; a hang or non-zero exit fails the WHOLE run (10s exit timeout), so do no expensive end-of-episode work. The protocol tolerates long first-frame stalls (a crewrift player took ~13s yet games ran), so a heavy synchronous startup dependency is acceptable behind a flag. Pin the protocol-authority runner.py to a commit SHA and auto-detect the first message shape (isinstance str => JSON player_config branch, else binary), because a runner version bump can instantly break a working player via a first-message mismatch -- symptom: all containers connect then disconnect at game start and `coworld play` hangs forever; check the manifest version before debugging your own logic.

#### 4. Player SDK architecture: layered stages, single-tick symbolic intents, typed-param leaves, single boundaries
`loop`

Factor a player into durable layers wired per-tick (perceive -> update_belief -> strategy.decide -> mode.decide -> resolve_action -> wire packets), reused across live and replay, enforced by an acyclic import chain (perception parses only, memory stores only, state reads memory, brain decides, pathfinding executes, telemetry records). Perception turns the raw frame (e.g. 128x128) into belief_state -- raw scene/sprite pixels must NEVER enter belief, the only interface strategy and modes see; when adding an LLM feed it this belief/evidence, not raw frames, and do not invent new perception primitives to justify the integration (localization, task detection, navigation, voting-UI parsing are already solved). Modes emit high-level SYMBOLIC intents only (navigate-to-X, complete-task; never button presses or multi-step interactions); make inner-loop granularity a single-tick action intent and push planning/pathing into declarative memory. Task primitives are pure frozen dataclasses with no mutable state; ALL multi-tick sequencing state lives on a separate ActionMemory. Leaf modes must NEVER read env/global config -- they receive typed params; the strategy reads config ONCE at the assembly boundary (build_runtime) and stamps params onto each directive (load-bearing where the framework rebuilds a fresh mode instance per activation: constructor injection silently breaks across the rebuild). Funnel state through one WorldSnapshot (co-equal to_dict()/to_prompt()) and decisions through one output command type the executor consumes blind to its source (making the LLM trivially optional). Keep the per-tick entry point thin and wire transport (send_input/send_chat) as injected callables, not an owned socket, so the core is testable without a live server. Model SDK-facing types as pydantic: FREEZE immutable per-tick values (percept, intent, command), leave the loop-mutated world model and action-execution state non-frozen, and use a plain dataclass only for numpy/bytes buffers that never reach the strategy. This layering recurs across players including the Orpheus/persephone framework.

#### 5. Split the policy into an LLM macro outer loop and a symbolic inner loop over a typed belief snapshot
`loop` · ⚠ _session-derived, unverified_

Make a curated typed belief snapshot (not raw observations) the API between loops, and never let the strategy/LLM outer loop emit raw game actions: its only output is a typed mode/directive selection while a local symbolic mode owns tick-by-tick action and movement. Scope the LLM to macro decisions only (mode, target, risk tolerance, companion/alibi target, kill permission, when to abandon a plan); this keeps the slow, latency-prone, unreliable LLM out of the per-tick hot path, confines its failures to mode selection, decouples per-game perception (OCR pixels vs grid vectors) from strategy so the outer loop and modes port across games, and cuts cost and game-losing structural mistakes. Distinguish durable gameplay directives (persist many ticks) from short situational meeting/burst actions and model them as different output types. Structure tunable prompts as three independent tiers selected at call time from context.self.role -- shared wire/output boilerplate, per-role goals, per-role strategy.

#### 6. Make the strategy<->runtime directive a strict typed contract with TTL, bounded knobs, and a safe inner fallback
`loop` · ⚠ _session-derived, unverified_

Define the directive as a frozen Pydantic model carrying mode/params/source/issued_at_tick/ttl_ticks/reason/metadata with typed bounded knobs (suspicion_threshold float [0,1], report_eagerness low|normal|high, voting_style evidence|majority|contrarian|skip_default), so invalid modes or wrong params are rejected before installation rather than blowing up mid-episode. Validate every ModeDirective in the ModeRegistry before installing it. Keep mode params as frozen extra-forbidden Pydantic ModeParams (never raw dicts); the inner loop and ModeRegistry must validate JSON into typed params before any mode sees it. Give parameterless modes (e.g. 'idle') a default-constructed params object via field(default_factory=ParamsType) -- on a frozen dataclass field(default_factory=...) is the correct pattern (a bare mutable default is rejected); give framework-managed objects a sane class-level default (params: ModeParams = ModeParams() on Mode); preserve positional-caller compatibility when adding a constructor arg by placing the new optional arg AFTER existing positional params, and prefer keyword args at policy decision points. Give the inner loop explicit reflexes plus a safe default and a TTL on every directive: the async/threaded strategy loop is slow, so if it stalls, errors, or goes silent the inner loop must degrade to a safe default rather than freeze on a stale directive. Feed the LLM enough context (mode params plus a scratch summary, not just mode name/source/ticks) to judge whether the current behavior is working and decide to abandon a failing plan.

#### 7. Exactly one active mode; reaffirm on matching directive, never reset live scratch; belief has one owner per namespace
`loop` · **negative result**

Keep exactly one active mode per agent: a deterministic symbolic local policy that reads belief, owns its own scratch, and emits a symbolic INTENT (navigate_to, complete_task, hunt, meeting) -- never raw transport commands. When a new directive matches the current mode's name AND params, REAFFIRM (preserve the live instance and scratch); only on a true change call on_exit/new instance/on_enter -- re-instantiating or resetting scratch on every reaffirmation destroys in-progress plans (a policy recognized 'we created this key whisper' on entry then recomputed and lost it next tick; PRESERVE the live instance when a directive re-affirms the same mode/params). Make mode switching a deliberate ordered sequence: validate directive against registry, run switch callbacks (which may override/cancel), clean up the old mode, activate the new, then run entry logic; doing these out of order (cleanup before validate) leaves the policy inconsistent. Give modes lifecycle methods mode_enter (setup only -- select_task runs immediately after) and mode_switch_cleanup (teardown by the outgoing mode). On finish/block return ModeDecision.complete()/.stalled(reason) so the runtime lowers the intent, installs the default directive, and wakes strategy with a reason; never let a stalled mode spin silently. Belief is the ONLY interface the strategy inspects -- keep raw frames/pixels out -- split into four owned namespaces (world truth: written only by perceive; mode scratch; action state; strategy inferences: written only via a named inference namespace) so there is exactly one owner of each mutation. #### 8. Mode lifecycle and task identity: structural equality, idempotent set_task, action memory survives strategic re-evaluation
`loop` · ⚠ _session-derived, unverified_

Define task identity as structural equality of (task_type, params) via frozen dataclasses, not object identity, and clear per-task scratch state (action memory: A* waypoints, button history) only when the task actually changes, never on mode switch -- action memory must survive strategic re-evaluation while inferences are replaced wholesale on switch (but on a strategy-mode switch decide per-state whether memory survives; defaulting to 'reset everything' silently discards useful cross-mode history). Make set_task idempotent on (task type, target): a no-op when both match the active task, but a different target overrides; otherwise re-issuing constructs a fresh instance that restarts a multi-frame motion from frame 0 and loops in a tiny arc. Ensure typical task durations fit the re-planning interval (a planner every ~2s/~48 frames truncates a motion task needing 72+ frames) or make tasks resumable, not restartable. Make task/action selection an explicit first-class return value (select_task(...) -> Task | None, None means keep current) over a delta the framework must parse. Build modes as self-contained registry-keyed subclasses each returning Task objects per tick; wire in stub (idle) modes early and replace stage-by-stage, keeping a new stage in its own module (advanced_modes.py) rather than extracting shared internals. Fold related sub-behaviors into one mode rather than over-fragmenting (a mining mode includes drop-off, targets a resource TYPE not a specific extractor). Route menu/multi-step UI navigation through a single shared navigator (MenuNavigator(steps).next_command(...)) rather than open-coding step counters per task, because multi-step sequences (category->item->target) are where off-by-one and missing-release-tick bugs hide. #### 9. Encode decision precedence as named gates with hysteresis, FSMs, cooldowns, and forced-ejection handling
`loop`

Encode decision precedence as named, ordered gates rather than nested conditionals, documenting why each bypass exists (phase overrides bypass per-role stubs; mode-completion bypasses the minimum-duration gate). Separate 'build derived StrategicState from belief' (role, team, room, urgency, objective, tick counters) from 'decide a directive off that state'. An outer-loop decision function that runs every tick will THRASH modes (the framework applies any differing directive), so implement hysteresis: a minimum dwell (>=48 ticks / 2s) unless a critical override fires (phase change, key objective completing, forced whisper), never interrupt an in-progress whisper, and store the FULL held directive (with strategic params) plus the last non-whisper directive in persistent belief_state.ext (not the wholesale-replaced inferences dict, which erases params); test both (no hysteresis dithers; no override loses to phase changes). Drive a mode's sub-protocol with an explicit FSM (named states ENTER/ASSESS/EVALUATE/EXIT, each mapping to exactly one symbolic task per tick) and treat 'no longer in the expected view but I did not initiate the exit' as a forced-ejection signal -> EXIT immediately, rebuild knowledge, complete with reason 'forced_ejection', since the environment can yank a mode out of a sub-state. Add action cooldowns and response timeouts as first-class state fields (an N-tick cooldown between menu ops, an M-tick peer-response timeout), because untethered per-tick interaction sub-protocols spam the game and stall. Make reflex-driven no-LLM switches revocable: give each reflex a TTL and a per-reflex cooldown (~96 ticks / ~4s at 24 FPS) so the LLM can override within the window without immediate re-firing. Build explicit stuck-state recovery (>120 idle ticks during Playing -> force scout; role-detection failure -> default to grunt and keep retrying).

#### 10. Provide liveness with reflexes, TTLs, watchdogs, and always-traced fallback reasons
`loop`

Guarantee liveness with TTLs, default directives, reflexes, and watchdogs. Reflexes are priority-ordered symbolic overrides running INSIDE the inner loop for urgent events that can't wait for the slow strategy pass (phase transitions, hazards, deadlines); higher priority wins, equal priority preserves registration order, and you must trace which rules were checked/won/skipped (emit a reflex_evaluated trace; implicit first-match-wins ordering is a hidden contract that breeds subtle bugs). Separate automatic reactive reflexes from LLM-directed strategic choices by time scale: a reflex ('body in view -> flee', fires automatically even for bodies you didn't make) versus a strategic follow-up ('after a confirmed kill, go be seen near a crewmate for an alibi'). On directive TTL expiry or active-mode is_legal(belief)==false, install the default directive -- but a silent fallback hides design bugs, so ALWAYS emit a trace reason. Add a watchdog/heartbeat counter so a wedged player (stuck mode, infinite task, dead WebSocket) is detectable rather than silently idle, and put the watchdog on the loop/pipeline object, not on belief state, so it survives resets. Add a ring buffer of the last N action commands to the standard inspectable fields so a mode can detect 'I'm sending the same command repeatedly' without bespoke per-task tracking. Give each reflex/interrupt an explicit cooldown (a concrete tick count pinned in the design doc) to prevent rapid re-firing and mode thrash.

#### 11. Separate intent from actuation and resolve all config once at the policy edge; inject, never poll env in leaves
`loop`

Keep ALL actuation -- movement, pathfinding, adjacent-cell interaction, button-pulse/cursor timing, menu open/close mechanics, multi-tick holds, movement-failure detection, transport packet/Action encoding -- in ONE action-resolver layer, and keep strategic priorities, long-horizon target choice, and social reasoning out of it: a mode emits a high-level discipline hint (DisciplineReport, DisciplineKillStrike vs DisciplineNormal, DisciplineTaskHold) and the action layer decides when to press (OR in button A only within action range), so button-timing logic lives in one place. Make the action layer stateful and intent-diffing: on unchanged intent continue executing (advance nav route, keep holding A, step the vote cursor), on changed intent discard in-progress execution and start fresh. Separate transport/platform config (runner env like COGAMES_ENGINE_WS_URL, read in policy_player.py) from behavior config (read in from_env) -- never mix them. Config leaks into behavior when an agent polls os.environ inside its decision loop (BE_DUMB/DICK_MODE every decision) or builds LLM clients from env at constructor time: resolve ALL configuration once at the composition root (build_runtime), reading and validating everything (fail-fast and loud rather than silently defaulting mid-run), and inject constructed clients/profiles/flags; leaves receive only piece-specific config (e.g. trace level), never raw config. Use a ModeRegistry.register_factory(name, factory, params_type=) seam so the composition root closes over injected dependencies (e.g. an llm_client) and builds composite modes, with register(cls) delegating to it; enforce factory return contracts in code (mode.name == registered_name plus a directive-match check inside create()), not just documented as MUST. Recognize an SDK DI seam does not by itself move strategy flags (dumb/dick) out of behavior logic; that needs a follow-up in the player's own config/strategy layer.

#### 12. Scope hooks at agent/mode levels with deterministic order and crash isolation; mutate live state back on rollback
`loop` · ⚠ _session-derived, unverified_

Scope player hooks at two levels: agent-level (always run) and mode-level (run only when a mode is active), with deterministic order (agent-level FIFO first, then current-mode FIFO, no interleaving). Wire explicit hook points at the loop's phase boundaries (e.g. post-belief-update) from the start so modes/logging/experiments observe or mutate state at known seams without forking the pipeline, making A/B instrumentation cheap. Make hook code defensively isolated: deep-copy the belief/game state before each call, and if a hook raises, restore the LIVE state in place (mutate fields back; do NOT rebind a fresh local copy, because callers and prior hooks hold references to the live instance), log, and keep the pipeline running so a buggy experimental hook never crashes the player mid-episode (assert by registering a deliberately crashing hook alongside a real one). Apply outer-loop inferences BEFORE firing mode-switch callbacks and let those callbacks override/veto the directive; keep mode-switch callbacks as a SEPARATE dispatch path from per-tick hooks because their signature differs (they may return an override directive). Order of operations inside a frame is load-bearing for observability: read transient per-frame wake/trigger flags BEFORE the step that clears them, because clearing wake-reason flags at frame end makes any tracing/consumer log empty/stale. Resolve ambiguous hook/callback signatures by deferring to the design doc's signature table verbatim rather than guessing.

#### 13. Cache the per-slot policy instance; rebuilding it per observation destroys all state

#### 14. LLM is an advisory layer on a deterministic floor: enforce legality in code, keep it off the per-tick loop
`loop` · ⚠ _session-derived, unverified_

Make the LLM advisory over a deterministic floor, never the per-tick actuator. Always compute the deterministic recommendation as fallback; enable the LLM for social/meeting chat and voting while keeping movement/mode transitions on the symbolic path, gated by explicit env flags (LLM_DISABLE, LLM_GAMEPLAY_DIRECTIVES). Never let it pick from the raw legal set: a deterministic ranker pre-ranks and passes only the top selectable action ids plus a briefing; validate the response against those ids under strict JSON; fall back to the pure ranker on disable/error/timeout/illegal-id. #### 15. LLM owns only the socially-complex surface via a closed schema; deterministic code owns everything mechanical
`loop`

Default to a HYBRID strategy: deterministic code computes affordances and hard constraints; the LLM selects among legal options using structured evidence and never presses buttons, never invents modes, falling back to deterministic policy when rejected. The LLM owns ONLY socially-complex surfaces (probe-target choice, reveal decisions, chat/deception wording and timing, coalition/vote choice). Make the boundary schema-driven with six pieces: (1) a context builder (belief -> compact JSON-safe context: identity, objective, player records with confidence/sources, recent messages, legal actions, affordances, cooldowns, failures, hard constraints); (2) a surface-specific prompt citing structured data, not raw frames; (3) a replaceable provider so deterministic fakes swap in; (4) a validator; (5) an executor mapping validated decision -> ModeDirective; (6) a full trace (context hash, raw decision, validation result, accepted directive, rejection reasons, latency, fallback). Move along the spectrum shadow -> advisory -> constrained -> full control one surface at a time, off by default, only after replay and trace review. Pair the LLM with a deterministic ranking DOCTRINE ported faithfully from the game image's own external-agent scorer (e.g. starter-framework.mjs rankLegalActions/actionScore) as a pure function so priorities match what the engine rewards; that doctrine both seeds the LLM's shortlist/briefing and serves as the fallback. A view-forced mode (e.g. in-whisper) can't be steered by a top-level decision alone -- expose it as a mode-local control surface consulting the LLM at specific decision points while it keeps its own timeouts/forced-exit safety.

#### 16. Gate every LLM action through a deterministic provider-free validator over a closed schema; never trust LLM memory
